# Implementation Plan: Multi-Ecosystem Expansion (Go, RPM, Maven, Cargo, Gem)

**Branch**: `003-multi-ecosystem-expansion` | **Date**: 2026-04-17 | **Spec**: [`spec.md`](./spec.md)
**Input**: Feature specification from `/specs/003-multi-ecosystem-expansion/spec.md`

## Summary

Add five new ecosystem readers to `mikebom sbom scan`, prioritised Go > RPM > Maven > Cargo > Gem. Go ships two sources — `go.mod`/`go.sum` for source trees and `runtime/debug.BuildInfo` for compiled binaries — closing the distroless/scratch image gap that currently restricts SBOM generation on those targets to syft and trivy. RPM closes the RHEL / Rocky / Fedora / Amazon Linux / CentOS / Oracle / Alma / SUSE compliance gap via a **pure-Rust** read of `/var/lib/rpm/rpmdb.sqlite`. Maven + Cargo + Gem round out the enterprise backend stack and leapfrog scalibr on three more dimensions. Every reader reuses the project-root walker, tier ladder, and dep-graph lookup infrastructure from milestone 002; no schema changes in `mikebom-common` are expected.

## Technical Context

**Language/Version**: Rust stable (same workspace compiler as milestones 001–002). No new nightly-only features required for user-space readers. `mikebom-ebpf` is untouched.
**Primary Dependencies (new)**:

- **`goblin`** (pure Rust) — ELF + Mach-O parsing for Go BuildInfo extraction. Rejects the `c` feature by default; we use no C.
- **`sqlite-parser`** style pure-Rust reader for `rpmdb.sqlite`. Evaluated in research.md (Principle I compliance: no `libsqlite3-sys`, no bundled C).
- **`zip`** (pure Rust) — JAR / WAR / EAR archive reading. Already available; no new C.
- **`quick-xml`** (pure Rust) — `pom.xml` parsing.
- **`toml`** — Cargo.lock parser. Already a workspace dep from milestone 002 (Poetry).
- **Custom line-parsers** — `Gemfile.lock`, `go.mod`, `go.sum` (all plain-text grammars, ~100 LOC each).
- **`packageurl`** (Rust) — unchanged from milestone 002; produces canonical PURLs that round-trip through `packageurl-python`.

**Storage**: N/A — all state is in-process for the duration of a single scan, same as milestone 002.
**Testing**: `cargo test --workspace` for unit + integration tests. Synthetic fixtures (file trees + docker-save tarballs built in-test) avoid checked-in binary blobs; fixtures that require real binaries (compiled Go programs) ship as checked-in `.bin` artifacts generated by a one-shot helper script documented in quickstart.md.
**Target Platform**: Linux (primary), macOS (CI). The scan pipeline is platform-agnostic; only Mode-1 trace is Linux-only. Go binary analysis must work on both ELF (Linux) and Mach-O (macOS) inputs.
**Project Type**: CLI tool — `mikebom sbom scan` subcommand, no new top-level subcommands in this milestone.
**Performance Goals**:

- SC-001 / SC-002 / SC-003 carried from spec (SBOM accuracy thresholds).
- SC-009: stripped Go binary up to 200 MB completes in <2 s.
- SC-010: polyglot directory with one project per ecosystem completes in <10 s.
- New per-image scaling guardrail (recommendation, not a hard SC): rpmdb.sqlite with ≥1000 rows parses in <500 ms on a modern laptop; an image with ≥100 JARs processes in <15 s.

**Constraints**:

- **Pure Rust, zero C** — Principle I. Binds the SQLite reader choice to a pure-Rust implementation.
- **No `.unwrap()` in production** — Principle IV. All new readers return `Result`; test-only code may use `.expect()`.
- **Offline support** — FR-007. Every ecosystem's core path is fully offline; `deps.dev` enrichment remains opt-in.
- **Hardened untrusted parsing** — FR-009 (resolved in clarification session). In-memory ZIP with zip-slip guards, read-only SQLite with per-query timeout + size cap, bounded ELF parsing.

**Scale/Scope**: ~2000–3000 LOC of new Rust in `mikebom-cli/src/scan_fs/package_db/` (five new modules) + ~500 LOC in the Go binary analyser + ~200 LOC in path_resolver + ~1500 LOC of tests + fixtures. Roughly twice the surface area of milestone 002.

**Strict Boundaries**:

1. **No lockfile-based dependency discovery** → interpreted as trace-mode-only per milestone 001 precedent. Scan mode has always read installed-package dbs and lockfiles for the components themselves. No change.
2. **No MITM proxy** → unaffected.
3. **No C code** → binding constraint for the rpmdb reader. Resolved in R1.
4. **No `.unwrap()` in production** → enforced per file. R10 adds a `cargo-clippy` gate `-D clippy::unwrap_used` to CI.
