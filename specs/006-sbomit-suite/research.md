# Phase 0 — Research: SBOMit compliance suite

Four open questions resolved before design. Each entry: **Decision**,
**Rationale**, **Alternatives considered**.

---

## R1. sigstore-rs suitability + dependency audit

**Decision**: Use the `sigstore` crate (the Rust SDK, crates.io
`sigstore` published by the sigstore organization). Wire it to handle
DSSE sign/verify, Fulcio-backed keyless cert issuance via OIDC, and
Rekor upload + inclusion-proof embedding.

**Rationale**:
- The `sigstore` crate is the canonical Rust implementation of the
  sigstore client protocol, actively maintained by the sigstore
  organization (recent activity every few weeks on GitHub).
- It supports all four required capabilities: (a) DSSE sign + verify;
  (b) Fulcio OIDC flow with built-in support for GitHub Actions, GitLab
  CI, and OAuth flows; (c) Rekor upload + inclusion proof retrieval;
  (d) local PEM key loading including sigstore's own encrypted key
  format and plain PEM.
- TLS: uses `rustls` by default (matches our existing
  `reqwest`-with-rustls posture from milestone 005). No `openssl-sys`
  in the critical path.
- Crypto primitives: `ring` (pure Rust, no C bindings despite the name
  — `ring` uses in-tree assembly for perf but no external C libs that
  link at build time), plus `ed25519-dalek`, `p256`, `rsa` — all pure
  Rust.

**Dependency audit** (Constitution Principle I gate): a `cargo tree -p
mikebom -e normal` after adding `sigstore` MUST show zero `*-sys` crates
that invoke a C compiler at build time. Known pitfalls:
- `openssl-sys`: if it appears, we've mis-configured `sigstore` or
  transitively pulled in a non-rustls HTTP client. Audit.
- `libz-sys` / `zlib-sys`: same treatment.
- `native-tls`: mikebom's existing `reqwest = { default-features =
  false, features = ["json", "rustls-tls"] }` pattern prevents this;
  verify `sigstore` doesn't re-enable `native-tls`.

Tentative deps entry for `mikebom-cli/Cargo.toml`:
```toml
sigstore = { version = "0.10", default-features = false, features =
    ["sigstore-trust-root", "cosign", "fulcio-native-tls-disabled",
     "rekor", "bundle"] }
```
(Exact feature names TBD during implementation; the principle is
"rustls, no native-tls, no openssl-sys".)

**Alternatives considered**:
- **Hand-rolled DSSE + ed25519 via `ed25519-dalek`**: would avoid the
  larger `sigstore` dep, but loses Fulcio/Rekor for free. We'd have to
  re-implement the OIDC flow, Fulcio cert parsing, and Rekor client —
  several thousand LOC of security-critical code. Rejected.
- **`cosign` as a shelled-out binary**: violates Principle I (C code
  in the pipeline; `cosign` is Go but still a binary dep), and makes
  the build non-hermetic. Rejected.
- **`in-toto-rs`**: narrower scope — does in-toto Statement
  serialization but not signing. Useful for layout work (R2) but not
  for signing.

**Fallback if audit fails**: if `sigstore` pulls in a C-linked crate
we can't feature-gate away, fall back to a combination of:
`dsse` (crates.io, pure Rust) + `ed25519-dalek` (pure Rust) + custom
Fulcio + Rekor clients over `reqwest` + rustls. Higher LOC, preserves
constitution compliance.

## R2. In-toto layout tooling

**Decision**: Build minimal layout generation + verification on top of
`serde_json`. Do not depend on an external in-toto layout crate.

**Rationale**:
- The in-toto layout spec is a well-defined JSON schema with ~6 top-
  level fields (`_type`, `expires`, `readme`, `keys`, `steps[]`,
  `inspect[]`). The parts mikebom needs are a strict subset:
  single-step layouts with one functionary key, one
  expected-artifact pattern, and no inspection rules.
- The closest Rust crate (`in-toto`) exists but is broader than we
  need, has a smaller maintenance cadence than `sigstore`, and pulls
  in its own crypto stack — which would duplicate what `sigstore`
  already provides.
- Building our own means ~200 LOC for layout emit + ~300 LOC for
  layout verification (functionary-key check, step-name match,
  expected-artifact pattern match). Reviewable in one PR; no
  constitutional concerns.

**Layout schema mikebom emits** (minimal shape):

```jsonc
{
  "_type": "layout",
  "expires": "2027-04-21T00:00:00Z",
  "keys": {
    "<keyid>": {
      "keytype": "ecdsa",
      "scheme": "ecdsa-sha2-nistp256",
      "keyval": { "public": "<PEM-encoded public key>" }
    }
  },
  "steps": [
    {
      "_name": "build-trace-capture",
      "expected_command": ["*"],
      "pubkeys": ["<keyid>"],
      "threshold": 1,
      "expected_materials": [],
      "expected_products": [
        ["MATCH", "*", "WITH", "PRODUCTS", "FROM", "build-trace-capture"]
      ]
    }
  ],
  "inspect": []
}
```

Verification logic:
1. Load the layout; verify its signature (if layouts are signed —
   initially they're unsigned drafts generated by `policy init`, can
   be extended to signed layouts later).
2. For each step, check that an attestation exists with the matching
   step name and a signature from one of the declared functionary
   keys.
3. Check that the attestation's subject(s) match the step's
   `expected_products` pattern.
4. Emit `VerificationReport::Pass` or
   `VerificationReport::Fail { mode: LayoutViolation, detail: "..." }`.

**Alternatives considered**:
- **Depend on `in-toto-rs`**: reviewed; crate exists but is ~3 years
  old (last published Oct 2023) and hasn't kept pace with the in-toto
  spec's layout-v2 evolution. Crate-in-hand risk > roll-our-own risk.
- **Shell out to `in-toto-verify` Python CLI**: violates Principle I
  (external toolchain) and breaks reproducible builds. Rejected.
- **Skip layout entirely in this feature, defer to later**: user story
  is P2 and the spec's primary gap-closure work doesn't strictly need
  layouts. But the user message ("full sbomit suite") explicitly
  includes layouts. Keep.

## R3. CycloneDX enrichment — JSON Patch format

**Decision**: Use **RFC 6902 JSON Patch** (the operation-list form:
`[{"op": "add", "path": "/components/3/licenses/-", "value": {...}}]`),
not JSON Merge Patch (RFC 7396).

**Rationale**:
- RFC 6902 is strictly more expressive for CycloneDX-style documents.
  The common enrichment operations — adding to a license list,
  appending to a properties array, setting a single supplier field —
  all fit cleanly in a sequence of `add` / `replace` / `test`
  operations.
- RFC 7396 (Merge Patch) can't append to arrays without replacing the
  entire array, which loses provenance for any existing items.
- RFC 6902 has a mature Rust implementation: `json-patch` on crates.io
  (pure Rust, depends only on `serde` + `serde_json` + `thiserror`).
  No C in the tree.
- Conflict semantics naturally fall out of the operation list:
  apply operations sequentially; an operation that depends on a
  previous operation's outcome will see it. When two independent
  patches are applied in sequence, the later one wins (last-write-wins
  at the operation level). This resolves the "enrichment patches that
  conflict" edge case deterministically.

Per-patch provenance metadata lives in the enriched SBOM as a new
property group:

```jsonc
"properties": [
  {
    "name": "mikebom:enrichment-patch[0]",
    "value": "{\"author\": \"security-team\", \"timestamp\": \"2026-04-21T...\",
              \"base_attestation\": \"sha256:...\", \"op_count\": 3}"
  }
]
```

This keeps enrichment discoverable by a verifier walking the SBOM
without inventing a new CycloneDX extension type.

**Alternatives considered**:
- **JSON Merge Patch**: simpler syntax, but can't append to arrays
  non-destructively. Reject.
- **Custom patch DSL**: domain-specific operators for "add license",
  "set supplier", etc. Over-engineering for the P3 story; RFC 6902
  covers 100% of the space already.
- **CycloneDX Extension (`externalReferences`)**: misuses a field
  designed for component-level metadata, not document-level
  enrichment provenance.

## R4. Fulcio / Rekor URLs + OIDC config

**Decision**:
- Default Fulcio URL: `https://fulcio.sigstore.dev` (sigstore
  public-good instance).
- Default Rekor URL: `https://rekor.sigstore.dev` (same).
- OIDC provider defaults to `https://oauth2.sigstore.dev/auth` for
  local dev (interactive OAuth2 flow); when GitHub Actions env vars
  are detected, use the `ACTIONS_ID_TOKEN_REQUEST_URL` + `_TOKEN`
  flow.
- `--fulcio-url` and `--rekor-url` flags let operators point at
  private sigstore instances.

**Rationale**:
- sigstore public-good instances are the de-facto default for
  `cosign` and every SBOMit-reference verifier. Matching them by
  default maximizes interop out of the box.
- GitHub Actions is the most common CI for this tool's likely users
  (per feature 006's explicit CI target). `sigstore-rs` has built-in
  support for detecting the Actions OIDC env vars.
- Private-instance flags give organizations an escape hatch for
  air-gapped or proprietary sigstore deployments without changing
  code paths.

**OIDC provider detection heuristic** (in `signer.rs`):

```
if env::var("ACTIONS_ID_TOKEN_REQUEST_URL").is_ok() &&
   env::var("ACTIONS_ID_TOKEN_REQUEST_TOKEN").is_ok() {
    OidcProvider::GitHubActions
} else if env::var("SIGSTORE_ID_TOKEN").is_ok() {
    OidcProvider::Explicit  // operator-supplied token
} else {
    OidcProvider::Interactive  // opens browser for OAuth2
}
```

For v1, only `GitHubActions` and `Interactive` modes are auto-routed;
other CI providers (GitLab, CircleCI, Buildkite) can be added per the
spec's assumption ("initial keyless signing support targets GitHub
Actions"). Other providers can use `SIGSTORE_ID_TOKEN` with a
pre-minted token from their own OIDC plumbing — covers the long tail
without per-provider code.

**Alternatives considered**:
- **Hardcoded sigstore URLs, no flags**: blocks any private-instance
  deployment. Reject; public-good defaults + override flags are
  standard practice.
- **Env-var-only OIDC config (no CLI flags)**: matches `cosign`'s
  approach more closely, but mikebom already has a CLI-flag-heavy
  convention per the noun-verb pattern. Flags win for discoverability
  via `--help`.
- **Auto-detect every known CI provider**: would require OIDC
  plumbing per provider. Defer to follow-ups; v1 covers GitHub
  Actions + explicit-token fallback, which handles 90% of realistic
  users.

## Summary of decisions

| Question | Decision |
|---|---|
| R1 | `sigstore` crate (with dep audit); fallback to `dsse`+`ed25519-dalek`+custom clients if audit fails |
| R2 | Roll-our-own minimal layout on `serde_json`; no external layout crate |
| R3 | RFC 6902 JSON Patch via `json-patch` crate; last-write-wins at operation level |
| R4 | Sigstore public-good defaults + `--fulcio-url` / `--rekor-url` overrides; GitHub Actions OIDC auto-detected |
