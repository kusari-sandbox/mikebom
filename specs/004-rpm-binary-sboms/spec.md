# Feature Specification: RPM Package-File Scanning & Generic Binary SBOMs

**Feature Branch**: `004-rpm-binary-sboms`
**Created**: 2026-04-18
**Status**: Draft
**Input**: User description — extend mikebom's `sbom scan` pipeline with two capabilities that current coverage lacks: (1) scan a standalone `.rpm` package artefact (not just the installed `rpmdb.sqlite`) and (2) emit an SBOM for a generic compiled binary that **isn't** a Go binary. Milestone 003 made Go-via-BuildInfo and rpmdb.sqlite work; this milestone closes the two matching artefact-level gaps — the `.rpm` file you download off a mirror, and the ELF / Mach-O / PE binary you got from vendor X that you don't otherwise have metadata for. (The milestone also adds opt-in legacy Berkeley-DB rpmdb support per the Q2 resolution, so pre-RHEL-8 / CentOS-7 / Amazon-Linux-2 rootfs scans become actionable rather than empty.)

## Clarifications

### Session 2026-04-18

- Q: Which binary formats must v1 of the generic-binary reader support? → **A: All three — ELF + Mach-O + PE (full cross-platform).** The generic-binary reader covers Linux (ELF), macOS (Mach-O), and Windows (PE) in v1. PE support adds IMPORT-table DLL linkage evidence on the same axis as ELF `DT_NEEDED` / Mach-O `LC_LOAD_DYLIB`; Authenticode / code-signing verification remains out of scope (separate provenance milestone).
- Q: Does this milestone also add legacy Berkeley-DB rpmdb (`/var/lib/rpm/Packages`) support? → **A: Yes, gated behind a `--include-legacy-rpmdb` opt-in flag.** Scans without the flag keep milestone-003 behaviour (diagnostic log, zero components); scans with the flag activate the BDB reader and produce rpm components from pre-RHEL-8 / CentOS-7 / Amazon-Linux-2 rootfs snapshots. The opt-in gate prevents silent behaviour change on existing BDB-containing scan targets.
- Q: How should a CGo Go binary's two readers (Go BuildInfo + generic ELF) compose their outputs? → **A: Option D — flat, cross-linked.** One file-level component per binary (`binary-class=elf`, `detected-go=true`); BuildInfo modules are emitted as top-level `pkg:golang/...` components; DT_NEEDED linkage evidence as top-level `pkg:generic/<soname>` components; neither is nested under the file-level component, both are cross-linked via `evidence.occurrences[]` that references the file-level component's BOM-ref. Matches the milestone-003 Go reader's existing top-level-modules shape and keeps the two axes independent for consumers.
- Q: Where does the embedded-version-string scanner look inside a binary, and what scoping controls false positives? → **A: Section-restricted — read-only string sections only.** Search is limited to format-appropriate read-only string containers: ELF `.rodata` + `.data.rel.ro`, Mach-O `__TEXT,__cstring` + `__TEXT,__const`, PE `.rdata`. Full-file `memmem` is explicitly rejected as prone to false positives from literal-string traffic in unrelated binaries. This keeps SC-005's zero-false-positive target achievable without sacrificing real-world detection — libraries embed self-identification as string literals, which always land in these sections.
- Q: How are linkage-evidence components deduplicated across multiple parent binaries in the same scan (e.g., ten ELFs all `DT_NEEDED`-ing `libssl.so.3`)? → **A: Dedup globally by PURL, merge occurrences.** The SBOM emits exactly one `pkg:generic/<soname-or-install-name-or-dll-name>` component regardless of how many parent binaries reference it; every parent binary path is recorded under that single component's `evidence.occurrences[]`. This matches FR-008's PURL-first dedup rule and keeps component counts honest (one "linkage" fact per unique library, not N copies).
- Q: Where in the CLI grammar does `--include-legacy-rpmdb` attach? → **A: Subcommand flag on `sbom scan` only.** Usage is `mikebom sbom scan --include-legacy-rpmdb [other flags]`; the flag is NOT a top-level global option. Matches existing milestone-002/003 conventions for `--offline` and `--include-dev`, keeps `sbom scan --help` as the single source of truth for scan-time options, and avoids confusing users with a pre-noun global that only takes effect inside one subcommand.
- Q: What evidence-kind values name the two rpmdb readers for consumer filtering? → **A: Canonicalize both paths — `rpmdb-sqlite` and `rpmdb-bdb`.** The milestone-003 sqlite reader is updated to emit `mikebom:evidence-kind = "rpmdb-sqlite"` as part of milestone 004; the new BDB reader emits `mikebom:evidence-kind = "rpmdb-bdb"`. Symmetric naming lets consumers filter for "anything from an installed rpmdb" via a simple `rpmdb-*` prefix match. The one-line sqlite update is treated as a thin milestone-004 housekeeping change, not a milestone-003 revision.

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Scan a standalone `.rpm` package file (Priority: P1) 🎯 MVP

A release engineer downloads `openssl-3.0.7-1.el9.x86_64.rpm` from a mirror before installing it and wants an SBOM for that package artefact itself — the name, version, release, epoch, arch, license, and declared `REQUIRES:` list — without installing it first. They run `mikebom sbom scan --path ./openssl-3.0.7-1.el9.x86_64.rpm` (or a directory containing one-or-more `.rpm` files) and get back a CycloneDX SBOM with one `pkg:rpm/<vendor>/<name>@<epoch>:<version>-<release>?arch=<arch>` component per `.rpm` file observed. When the `.rpm` file's header carries a `Vendor:` tag (e.g., `Red Hat, Inc.` / `Fedora Project`), the vendor segment derives from that tag; otherwise the scanner falls back to an OS-release probe on the same rootfs (if any) and finally to `rpm` if neither is present.

**Why this priority**: This is the matching half of milestone 003's rpmdb work. rpmdb.sqlite answers "what's installed"; a `.rpm` file answers "what's in this artefact before install" — the question release engineers, mirror maintainers, and air-gapped deployment teams actually ask. Tools like syft, trivy, and RPM's own `rpm -qip` handle it; mikebom today returns zero components when pointed at a `.rpm`. This is a reputational gap on RPM as much as a functional one.

**Independent Test**: Place a curated set of at least five real `.rpm` files (small ones — e.g. `openssl-libs`, `zlib`, `glibc`, `bash`, `coreutils` from a known RHEL 9 / Fedora / Rocky mirror) in a fixtures directory. Scan the directory. Verify each produces exactly one component with canonical PURL, the correct license, the correct `REQUIRES:` list (tokenised to bare package names) as dependency edges where the target is observed in the same scan, and that the entries are deduplicated against any rpmdb.sqlite entries (if both are present).

**Acceptance Scenarios**:

1. **Given** a `.rpm` file named `openssl-libs-3.0.7-28.el9_4.x86_64.rpm`, **When** the user runs `mikebom sbom scan --path <the-file>`, **Then** the SBOM contains exactly one component with PURL `pkg:rpm/redhat/openssl-libs@3.0.7-28.el9_4?arch=x86_64` (epoch `0` omitted per packageurl-python canonical form), the license field is populated from the RPM header's `License` tag, and the `supplier.name` is populated from the RPM header's `Vendor` tag.
2. **Given** a directory containing ten `.rpm` files from a mixed set of vendors (RHEL, Fedora, Rocky), **When** the user scans the directory, **Then** every `.rpm` surfaces as exactly one component with its vendor-appropriate PURL slug (`redhat` / `fedora` / `rocky`) derived from the header `Vendor:` tag through the existing milestone-003 `rpm_vendor_from_id` mapping.
3. **Given** a `.rpm` file whose header declares `Requires: openssl-libs libc.so.6 coreutils`, **When** the scan runs and another `.rpm` named `openssl-libs-3.0.7-28.el9_4.x86_64.rpm` is also present in the scan root, **Then** the SBOM's `dependencies[]` section contains an edge from the parent package to `openssl-libs` (dangling targets like `libc.so.6` drop silently — they're soname provides, not package requires — and `coreutils` drops because no matching `.rpm` is observed).
4. **Given** a rootfs that contains BOTH `/var/lib/rpm/rpmdb.sqlite` (with openssl-libs installed) AND a dropped-in `/tmp/openssl-libs-3.0.7-28.el9_4.x86_64.rpm`, **When** the user scans that rootfs, **Then** `openssl-libs` surfaces exactly once — deduplicated by PURL — and the winning source is the installed rpmdb entry (it's ground truth for "installed on this system"); the `.rpm`-file evidence is merged as a supplementary source.
5. **Given** a malformed `.rpm` file (truncated header, wrong magic, or unreadable lead block), **When** the user scans its directory, **Then** the scan completes without error, emits zero components for that file, and logs exactly one WARN line naming the offending file — fail-graceful per the milestone-003 defense-in-depth baseline.

### User Story 2 — Scan a generic (non-Go) ELF binary with embedded library evidence (Priority: P2)

A security engineer is handed a single stripped ELF binary from a vendor — no source, no container, no package metadata. They want to know what it dynamically links against and whether any well-known embedded libraries are detectable. They run `mikebom sbom scan --path ./vendor-tool` and get back an SBOM that:

1. Lists the binary itself as a file-level component with hashes and a `mikebom:binary-class` property set to `"elf"` / `"macho"` / `"pe"` as appropriate.
2. Lists each dynamically-linked library referenced by `DT_NEEDED` (ELF), `LC_LOAD_DYLIB` (Mach-O), or the PE IMPORT / Delay-Load IMPORT directories (PE) as a **linkage evidence** component — identified by soname / install-name / DLL-name only, since the binary alone can't tell us the installed version.
3. Lists embedded library self-identifications — via `ELF .note.package` (Fedora/Debian distro-encoded identity), and via a small curated set of well-known self-identifying version strings (OpenSSL's `OpenSSL <ver>`, zlib's `deflate <ver>`, SQLite's `3.<ver>`, curl's `libcurl/<ver>`) — as `analyzed`-tier components with PURLs only when the match is unambiguous.
4. Flags stripped binaries with `mikebom:binary-stripped = true` so the transparency layer informs the consumer that the scan was constrained.

**Why this priority**: This is the generic-binary analogue of Go's BuildInfo path. Every other mainstream scanner (syft, trivy, scalibr) does at least some form of non-Go binary analysis — dynamic-linkage and/or embedded-string heuristics — because real containers and real servers carry plenty of binaries that aren't Go. Today mikebom emits nothing for a non-Go ELF. Closing this gap (across all three mainstream executable formats per the Q1 resolution) is the final piece of "we do binary SBOMs, not just language-ecosystem SBOMs" positioning. P2 below .rpm because RPM package-file scanning has a harder correctness ceiling (we can be fully right) while generic binary analysis is inherently heuristic (we can only ever be partially right), and we want to set that expectation via transparency rather than by overselling coverage.

**Independent Test**: Supply a curated fixtures set covering all three executable formats — (a) a dynamically-linked `busybox`-style ELF, (b) a Mach-O binary from `/usr/bin/` on macOS (`LC_LOAD_DYLIB` entries), (c) a Windows PE / DLL with a populated IMPORT table (can be produced by MinGW or cross-compiled with Rust's `x86_64-pc-windows-gnu` target), (d) an ELF binary with a `.note.package` section (synthesised via `objcopy --add-section` in the test harness), (e) an ELF with an embedded `OpenSSL 3.0.11` version string (`strings` visible), (f) a stripped ELF with no evidence at all. Verify: the dynamic-linked ELF emits one file-level component + one linkage-evidence component per `DT_NEEDED`; the Mach-O emits one per `LC_LOAD_DYLIB`; the PE emits one per IMPORT-table DLL reference; (d) emits a genuine PURL from the `.note.package` contents; (e) emits an OpenSSL component tagged `mikebom:evidence-kind = "embedded-version-string"`; (f) emits only the file-level component with `mikebom:binary-stripped` and an empty linkage list.

**Acceptance Scenarios**:

1. **Given** a dynamically-linked ELF binary with `DT_NEEDED` entries `libc.so.6`, `libssl.so.3`, `libcrypto.so.3`, **When** the user scans it, **Then** the SBOM contains a file-level component for the binary plus three linkage-evidence components keyed by soname, each tagged `mikebom:evidence-kind = "dynamic-linkage"` and `mikebom:sbom-tier = "analyzed"`.
2. **Given** an ELF binary whose `.note.package` section contains the distro-encoded JSON payload `{"type":"rpm","name":"curl","version":"8.2.1","architecture":"x86_64","distro":"Fedora","osCpe":"cpe:/o:fedoraproject:fedora:39"}`, **When** the user scans it, **Then** the SBOM contains a component with PURL `pkg:rpm/fedora/curl@8.2.1?arch=x86_64` tagged `mikebom:evidence-kind = "elf-note-package"` and `mikebom:sbom-tier = "source"` (distro self-identification is ground truth).
3. **Given** a statically-linked ELF with a readable `OpenSSL 3.0.11 19 Sep 2023` string at a stable offset, **When** the user scans it, **Then** the SBOM contains a `pkg:generic/openssl@3.0.11` component tagged `mikebom:evidence-kind = "embedded-version-string"` and `mikebom:confidence = "heuristic"`.
4. **Given** a Mach-O binary with `LC_LOAD_DYLIB` entries for `/usr/lib/libSystem.B.dylib` and `@rpath/libssl.48.dylib`, **When** the user scans it, **Then** the SBOM contains two linkage-evidence components keyed by install-name, identical treatment to ELF `DT_NEEDED`.
5. **Given** a Windows PE executable or DLL whose IMPORT table references `KERNEL32.dll`, `ADVAPI32.dll`, and `libssl-3-x64.dll`, **When** the user scans it, **Then** the SBOM contains a file-level component with `mikebom:binary-class = "pe"` plus three linkage-evidence components keyed by the referenced DLL name (`pkg:generic/<dll-name>`), each tagged `mikebom:evidence-kind = "dynamic-linkage"` and `mikebom:sbom-tier = "analyzed"`.
6. **Given** a file that is not a recognised executable binary (e.g., a Python bytecode file, a shell script, a text document), **When** the user scans the parent directory, **Then** the scanner skips it cleanly — no component emitted — with no WARN log, because it's not a binary format the reader claims to handle.
7. **Given** a Go binary that also has `DT_NEEDED` entries (CGo-linked Go is common), **When** the user scans it, **Then** the SBOM emits exactly ONE file-level component for the binary (carrying `mikebom:binary-class = "elf"` AND `mikebom:detected-go = true`); BuildInfo modules appear as top-level `pkg:golang/<module>@<version>` components (source-tier); DT_NEEDED linkage evidence appears as top-level `pkg:generic/<soname>` components (analyzed-tier); neither set is nested under the file-level component — both sets' `evidence.occurrences[]` reference the file-level component's BOM-ref for traceability.

### User Story 3 — Polyglot scan combining `.rpm` files and generic binaries (Priority: P3)

A platform-security team has a directory of air-gapped deployment artefacts: ten `.rpm` files bundled next to a handful of vendor-supplied statically-linked ELF binaries. They run `mikebom sbom scan --path <artefact-dir>` once and get one CycloneDX SBOM covering the full set — RPM components from the `.rpm` headers, plus binary components from the ELFs. Every component carries its evidence-kind and tier so the consumer can judge trust per-entry.

**Why this priority**: Ties P1 and P2 together, validates that the composition records are right across both ecosystems simultaneously, and matches the real-world pattern of how these two artefact types are delivered — mixed in the same directory, not siloed. Demoable in a single invocation.

**Independent Test**: Place five `.rpm` files plus a representative set spanning all three executable formats (three ELF + one Mach-O + one PE) into a tempdir. One scan. Verify the SBOM's `components[]` contains entries from every evidence-kind, that PURL formats are valid per packageurl-python for both, that the `rpm` composition record is `aggregate: incomplete_first_party_only` (we have only the `.rpm` files, not an installed rpmdb), and that no `aggregate` record exists for the generic-binary ecosystem (it's always heuristic — never complete).

**Acceptance Scenarios**:

1. **Given** a directory with five `.rpm` files and five binaries spanning ELF + Mach-O + PE, **When** the user runs `mikebom sbom scan --path <dir>`, **Then** the resulting SBOM contains ≥5 `pkg:rpm/...` components AND ≥5 file-level binary components — one per binary, with the correct `mikebom:binary-class` value per format — in one output document.
2. **Given** the same directory, **When** the scan completes, **Then** the `compositions[]` section contains an `rpm` entry with `aggregate: incomplete_first_party_only` (because `.rpm` files are not an installed-package-db view) but contains no `aggregate: complete` entry anywhere for generic-binary evidence.

### User Story 4 — Scan a legacy RHEL 7 / CentOS 7 / Amazon Linux 2 image via the BDB rpmdb (Priority: P3)

A compliance-driven platform team still operates RHEL 7 and Amazon Linux 2 images that use the pre-sqlite Berkeley-DB rpmdb (`/var/lib/rpm/Packages`). They have previously been unable to SBOM-scan those images with mikebom. With this milestone they run `mikebom sbom scan --image <amzn2-image>.tar --include-legacy-rpmdb` and get an SBOM listing every installed package — same shape as a RHEL 9 scan — from the Berkeley-DB-backed rpmdb.

**Why this priority**: RHEL 7 reaches official vendor EOL in 2024 but remains in active use in long-lived regulated environments (financial services, public sector, industrial control systems) well past that date; Amazon Linux 2 is still a supported AL runtime for many workloads. Without a BDB reader, mikebom emits zero rpm components for those images — the same gap milestone 003 closed for sqlite-backed rpmdbs. This milestone closes the final rpmdb gap behind the opt-in flag, which avoids a surprising behaviour change on pre-existing scans that today legitimately return zero components.

**Independent Test**: Scan a rootfs snapshot of a minimal Amazon-Linux-2 or CentOS-7 image twice — once without `--include-legacy-rpmdb` (expect: milestone-003 behaviour, one WARN log line pointing at the legacy BDB path, zero rpm components) and once with the flag (expect: component count matches what `rpm -qa | wc -l` would report, every PURL round-trips, every component carries license + dependencies).

**Acceptance Scenarios**:

1. **Given** a rootfs containing `/var/lib/rpm/Packages` (BDB format) but no `rpmdb.sqlite`, **When** the user runs `mikebom sbom scan --path <rootfs>` **without** `--include-legacy-rpmdb`, **Then** the scan completes with a single WARN log line and zero rpm components — identical to milestone-003 behaviour.
2. **Given** the same rootfs, **When** the user runs `mikebom sbom scan --path <rootfs> --include-legacy-rpmdb`, **Then** the SBOM contains one `pkg:rpm/<vendor>/<name>@<epoch>:<version>-<release>?arch=<arch>` component per installed package; licenses, `supplier.name`, and `REQUIRES:`-derived dependency edges populate the same way as the sqlite-backed path.
3. **Given** a rootfs that contains BOTH `/var/lib/rpm/rpmdb.sqlite` AND `/var/lib/rpm/Packages` (a transitional configuration), **When** `--include-legacy-rpmdb` is set, **Then** the sqlite path wins (it's the newer format) and the BDB reader is not activated; when the flag is unset, behaviour is unchanged from milestone 003.
4. **Given** a malformed BDB rpmdb (corrupt page header, invalid hash-page metadata, HeaderBlob region over-cap), **When** `--include-legacy-rpmdb` is set, **Then** the scan completes with zero rpm components and exactly one WARN log line naming the file — fail-graceful, same contract as rpmdb.sqlite.

### Edge Cases

- **`.rpm` file with no `Vendor:` tag** (some community rebuilds strip it): fall back to `/etc/os-release` on the same rootfs if any, then to `rpm` as the final fallback. Record the fallback via `mikebom:vendor-source = "os-release"` / `"fallback"`.
- **`.rpm` file with a detached signature** (`.rpm.asc` alongside): scan emits the component; signature verification is out of scope for this milestone (see Out of Scope).
- **`.rpm` file using XZ-compressed payload with zero package files** (a metapackage like `centos-release`): parse the header, emit the component; zero package files is valid.
- **Two `.rpm` files with the same PURL** (duplicate downloads): dedup by PURL; keep the first-observed physical path as the evidence.
- **A `.rpm` file that's actually a SRPM** (`.src.rpm`): emit the component with `arch=src` per packageurl-python canonical form; do NOT walk the embedded source tarballs.
- **`.rpm` with a `%pre` / `%post` script that declares transitive requires via `rpm -q --scripts`**: out of scope. We read only `REQUIRES:` and `PROVIDES:`.
- **ELF binary with `DT_NEEDED` entries but no ELF-note-package and no recognised version string**: emit only the file-level component + linkage evidence; no fabricated "this is libssl" identification without corroborating evidence.
- **ELF binary compiled with musl vs glibc**: both surface as linkage evidence for `libc.so.6` (glibc) vs `libc.musl-<arch>.so.1` (musl); the soname IS the evidence, we don't try to disambiguate beyond that.
- **Mach-O universal (fat) binary**: scan each architecture slice; emit linkage evidence merged across slices (soname + install-name are architecture-invariant in practice).
- **Binary compressed with UPX**: emit the file-level component; unpack UPX is out of scope. Record `mikebom:binary-packed = "upx"` when the UPX signature is detected so the consumer knows the linkage list is artificially sparse.
- **Binary that's ELF-valid but statically linked and stripped**: emit the file-level component only — no linkage evidence, no version strings; tag `mikebom:binary-stripped = true` and `mikebom:linkage-kind = "static"`.
- **Binary with ELF-note-package declaring `type: alpm`** (Arch Linux): the reader emits `pkg:alpm/...` PURLs when the type segment is one of the packageurl-known ecosystems. Unknown `type` values emit `pkg:generic/...` with the raw type recorded in `mikebom:elf-note-package-type`.
- **`.rpm` file inside a tarball**: not walked. Tar inspection is a separate capability and out of scope here.
- **`.rpm` file on a case-insensitive filesystem named `OPENSSL-LIBS-....RPM`**: extension match is case-insensitive (`.rpm` / `.RPM` both match); mirrors deb / apk file-readers' behaviour.
- **PE binary with a `.NET` / CLI IMPORT table (`mscoree.dll` only)**: emit the file-level component plus the single linkage-evidence entry for `mscoree.dll`. Deeper .NET introspection (CLI metadata tables for assembly references) is out of scope — the IMPORT table captures what the unmanaged loader sees.
- **PE binary with a Delay-Load IMPORT directory**: parse it on the same axis as the regular IMPORT directory; emit linkage-evidence entries. Bound-imports directory is redundant and skipped.
- **PE binary with an Authenticode signature block in the certificate table**: the bytes are skipped; the reader does NOT verify or extract the signing chain (Out of Scope).
- **BDB rpmdb on a filesystem where `/var/lib/rpm/Packages` is a symlink to an absolute host path outside the scan root** (unusual, but seen on some image-builder outputs): the reader refuses to follow the symlink out of the scan root; logs a WARN; emits zero components — same containment behaviour as the milestone-003 defense-in-depth baseline.
- **BDB rpmdb smaller than one page** (empty / never-initialised): reader treats it as a malformed file — single WARN, zero components.
- **`--include-legacy-rpmdb` set against a rootfs with only an sqlite rpmdb** (the flag is a no-op here): behaviour is unchanged — sqlite path runs as normal, the flag is silently ignored. The flag gates BDB activation only, not overall rpm behaviour.

## Requirements *(mandatory)*

### Functional Requirements

#### Common to this milestone

- **FR-001**: The scan_fs pipeline MUST discover `.rpm` files and generic binaries via the same bounded recursive walk used for other ecosystem readers in milestones 002 / 003 — no new walker primitive, only new per-file matchers.
- **FR-002**: Every PURL emitted by either new reader MUST match the canonical encoding of the packageurl-python reference implementation. A round-trip conformance test is mandatory per evidence-kind.
- **FR-003**: Every new component MUST carry `mikebom:sbom-tier` per the R13 ladder established in milestone 002 — `source` for RPM-header and ELF-note-package evidence, `analyzed` for heuristic binary evidence (DT_NEEDED / LC_LOAD_DYLIB / PE-imports / embedded version strings), `deployed` for BDB-rpmdb entries (same tier as the sqlite-rpmdb reader in milestone 003 — installed-db evidence is ground-truth-for-what's-installed regardless of which on-disk format stored it).
- **FR-004**: Every new component MUST carry `mikebom:evidence-kind` identifying how it was discovered. Enumerated values in v1: `rpm-file` (the `.rpm` artefact reader), `rpmdb-sqlite` (the milestone-003 sqlite rpmdb reader — updated as part of this milestone to emit this canonical value), `rpmdb-bdb` (the legacy BDB rpmdb reader), `dynamic-linkage` (ELF DT_NEEDED / Mach-O LC_LOAD_DYLIB / PE IMPORT table), `elf-note-package`, `embedded-version-string`. Unlisted values are a spec violation. The `rpmdb-sqlite` and `rpmdb-bdb` pair share a `rpmdb-*` prefix so consumers can filter for "any installed-rpmdb evidence" regardless of on-disk format.
- **FR-005**: Both readers MUST be a no-op when the scan root contains no `.rpm` files and no binaries of the supported formats. Empty scan is never an error.
- **FR-006**: Both readers MUST respect the global `--offline` flag (trivially true — neither reader needs the network to emit a valid SBOM; no deps.dev enrichment is part of MVP for either).
- **FR-007**: Both readers MUST apply the milestone-003 defense-in-depth baseline on untrusted input:
    - **`.rpm` files**: enforce a per-file size cap (recommend ≤200 MB; anything bigger is almost certainly not a real RPM), validate the lead-block magic (`\xED\xAB\xEE\xDB`) before parsing, bound the header-index-entry count to prevent header-bomb DoS, and read header-value bytes from a memory-mapped region bounded by the declared header size.
    - **Binary files**: bound parsed-section-header count, bound string-table reads by the declared table size, refuse files smaller than 1 KB or larger than 500 MB (mirrors Go binary path), validate magic before parsing.
    - **Transparency**: any reader that hits its defense-in-depth cap emits a file-level component with `mikebom:binary-parse-limit = "<reason>"` so the consumer sees the gap.
- **FR-008**: New components MUST dedup against existing readers' output by PURL — a package observed in both rpmdb.sqlite AND a sibling `.rpm` file emits one component with both evidence-kinds merged into `evidence.occurrences[]`; a Go binary's BuildInfo modules AND the same binary's DT_NEEDED linkage evidence are not conflated (different kinds of facts, different axes) and emit side-by-side components.

#### RPM `.rpm` file reader (US1)

- **FR-010**: The `.rpm` reader MUST parse the RPM v3 / v4 package-file format — lead block (96 B), signature header (unused for identity), main header (name, epoch, version, release, arch, license, vendor, packager, requires, provides, summary, description). The reader MUST be pure Rust; no shell-out to `rpm2cpio` / `rpm`. Dependencies are limited to crates already in the workspace or a small-surface `rpm` crate if constitution-vetted (Principle I).
- **FR-011**: The `.rpm` reader MUST match files by extension `*.rpm` (case-insensitive) AND by magic-byte-match at offset 0 (`\xED\xAB\xEE\xDB`). Extension-match alone is insufficient — someone renames a file; the reader must check the magic.
- **FR-012**: The `.rpm` PURL MUST follow the canonical form `pkg:rpm/<vendor>/<name>@<epoch>:<version>-<release>?arch=<arch>` — same shape as milestone 003's rpmdb path. Epoch `0` is omitted.
- **FR-013**: The `.rpm` reader MUST derive the PURL `<vendor>` segment via this priority order: (1) the RPM header's `Vendor:` tag mapped through a lookup table (`Red Hat, Inc.` → `redhat`, `Fedora Project` → `fedora`, `Rocky Enterprise Software Foundation` → `rocky`, `Amazon Linux` → `amazon`, `CentOS` → `centos`, `Oracle America` → `oracle`, `AlmaLinux OS Foundation` → `almalinux`, `SUSE LLC` → `suse`, `openSUSE` → `opensuse`), then (2) fall back to `/etc/os-release::ID` on the same rootfs if present (reusing milestone-003's `rpm_vendor_from_id`), then (3) fall back to the string `rpm`. The chosen source MUST be recorded on the component via `mikebom:vendor-source` (`header` / `os-release` / `fallback`).
- **FR-014**: The `.rpm` reader MUST populate component fields from header tags verbatim: license → `licenses[]` (fed through the existing SPDX canonicaliser), vendor → `supplier.name`, packager → `author.name`, summary + description → `description`.
- **FR-015**: The `.rpm` reader MUST tokenise the `REQUIRES:` tag to bare package names (drop version constraints — `openssl-libs >= 3.0` becomes `openssl-libs`) and emit dependency edges via the existing scan_fs edge resolver when the requirement target is observed elsewhere in the same scan (as a `.rpm` file OR as an rpmdb entry OR in `PROVIDES:` of another observed `.rpm`). Dangling requirements drop silently.
- **FR-016**: The `.rpm` reader MUST detect SRPMs (`arch == "src"` in the header) and emit `arch=src` in the PURL per packageurl-python canonical form; the reader MUST NOT attempt to walk the embedded source tarball (those payloads are out of scope).
- **FR-017**: When a `.rpm` is malformed (bad magic, truncated header, header-index count over cap), the reader MUST emit zero components for that file and log exactly one WARN line naming the file. The milestone-003 rpmdb "fail graceful, don't fail the scan" policy applies verbatim.

#### Legacy Berkeley-DB rpmdb reader (US4 — opt-in)

- **FR-018**: The scanner MUST add a `--include-legacy-rpmdb` CLI flag implemented as `global = true` on the top-level `Cli` struct (matching the existing `--offline` and `--include-dev` conventions). Canonical documented invocation is `mikebom sbom scan --include-legacy-rpmdb …`; the pre-noun position `mikebom --include-legacy-rpmdb sbom scan …` also works because clap's `global = true` accepts both positions — this is incidental and not the advertised shape. The flag MUST also be exposable via the `MIKEBOM_INCLUDE_LEGACY_RPMDB=1` env var to match existing env-toggle conventions. When the flag is unset, milestone-003 behaviour is preserved: a WARN diagnostic is logged if `/var/lib/rpm/Packages` is observed without `rpmdb.sqlite`, zero rpm components are emitted from the BDB path, and the sqlite path is unaffected. When the flag is set AND `rpmdb.sqlite` is absent AND `/var/lib/rpm/Packages` is present, the BDB reader activates and produces `PackageDbEntry` rows in the same shape the sqlite reader produces.
- **FR-019**: The BDB reader MUST be pure Rust; no shell-out to `rpm` / `rpmdb_dump` / `db_dump` / libdb. The implementation MAY use a vetted existing crate if constitution-clean OR an in-house subset reader (~600–900 LOC) covering Hash and BTree page layouts and the HeaderBlob record format (same shape the sqlite reader already parses from its row payloads). Overflow pages, master password, and the transaction log are out of scope; unreadable inputs emit a single WARN and zero components.
- **FR-019a**: BDB components MUST populate identical fields to the sqlite-path components (name, epoch, version, release, arch, license, vendor, packager, requires, provides). PURLs MUST pass the same round-trip packageurl-python conformance. The `mikebom:evidence-kind` value on BDB components is `rpmdb-bdb`; the milestone-003 sqlite reader is updated in this milestone to emit `rpmdb-sqlite` (per Q7 clarification), so the two installed-rpmdb paths share a canonical `rpmdb-*` namespace and consumers can filter-by-prefix.
- **FR-019b**: The BDB reader MUST apply the same defense-in-depth baseline as the rpmdb.sqlite reader — per-file size cap (recommend ≤200 MB), bounded record-count iteration with wall-clock budget (recommend ≤2 s), and zero-components-on-error rather than propagating failures upward.
- **FR-019c**: When both `rpmdb.sqlite` AND `/var/lib/rpm/Packages` exist on the same rootfs, the sqlite reader wins and the BDB reader is skipped regardless of `--include-legacy-rpmdb`. This prevents double-counting during filesystem transitions (some image-builder outputs ship both).

#### Generic binary reader (US2)

- **FR-020**: The generic-binary reader MUST detect and parse all three major executable formats: ELF (Linux), Mach-O (macOS), and PE (Windows). The `object` crate (already in the workspace) covers all three parsers with no C code, satisfying Principle I. Detection is magic-byte based at offset 0: `\x7FELF` (ELF), `\xFE\xED\xFA\xCE` / `\xFE\xED\xFA\xCF` / `\xCF\xFA\xED\xFE` / `\xCE\xFA\xED\xFE` / `\xCA\xFE\xBA\xBE` (Mach-O incl. fat), `MZ` at 0 with a valid `PE\0\0` signature at the offset stored in the DOS header (PE).
- **FR-021**: The generic-binary reader MUST emit a file-level component per binary with: file hashes (SHA-256 + SHA-1 per Principle XI), `mikebom:binary-class` (one of `elf` / `macho` / `pe`), `mikebom:binary-stripped` (bool — true when the format-appropriate symbol/metadata tables are absent), `mikebom:linkage-kind` (`dynamic` / `static` / `mixed`), and when detected, `mikebom:binary-packed = "upx"`.
- **FR-022**: For ELF binaries, the reader MUST extract `DT_NEEDED` entries from `.dynamic` and emit one linkage-evidence component per entry keyed by soname. The PURL form is `pkg:generic/<soname>` (no version — the soname alone is evidence; version must come from something else). Add `mikebom:evidence-kind = "dynamic-linkage"` and record the parent binary's relative path via `evidence.occurrences[]`.
- **FR-023**: For Mach-O binaries, the reader MUST extract `LC_LOAD_DYLIB` install-names and emit linkage-evidence components keyed by the full install-name. For fat (universal) Mach-O, the reader iterates every architecture slice and emits one merged linkage-evidence set (deduplicated by install-name across slices — they are architecture-invariant in practice). Same PURL shape and evidence metadata as ELF.
- **FR-023a**: For PE binaries, the reader MUST extract referenced DLL names from the IMPORT directory AND the Delay-Load IMPORT directory, and emit one linkage-evidence component per unique DLL name. The PURL form is `pkg:generic/<dll-name>` (no version) — parallel to ELF soname / Mach-O install-name. The `mikebom:evidence-kind` value is `dynamic-linkage` (shared with ELF and Mach-O — the axis is the same, only the format differs). The `mikebom:binary-class` property distinguishes them. Bound-imports and the Authenticode certificate table are skipped.
- **FR-024**: For ELF binaries with a `.note.package` section, the reader MUST parse the distro-encoded payload (systemd's `Packaging Metadata Notes` format — JSON with `type`, `name`, `version`, `architecture`, `distro`, `osCpe`). When `type` is one of `rpm` / `deb` / `apk` / `alpm` / `pacman` / `pkg`, emit a PURL of that ecosystem's canonical form (vendor derived from `distro` via the milestone-003 mapping table). Other `type` values emit `pkg:generic/<name>@<version>` with the raw type preserved via `mikebom:elf-note-package-type`.
- **FR-025**: The embedded-version-string detector MUST match against a **bounded curated set** of well-known self-identifying libraries at v1 — minimum set: OpenSSL (`OpenSSL <ver>`), zlib (`deflate <ver>`), SQLite (`3.<ver>`), curl (`libcurl/<ver>`), PCRE (`<ver>\0PCRE`). The set is defined in code, not user-configurable, to keep the false-positive surface small. The scanner MUST restrict its search to format-appropriate read-only string sections — ELF `.rodata` and `.data.rel.ro`, Mach-O `__TEXT,__cstring` and `__TEXT,__const`, PE `.rdata` — and MUST NOT run against the full binary image (full-file `memmem` is explicitly rejected per Q4 clarification; it generates false positives on any binary whose literal-string traffic mentions these libraries). Components emitted via this path MUST be tagged `mikebom:evidence-kind = "embedded-version-string"` AND `mikebom:confidence = "heuristic"`.
- **FR-026**: The generic-binary reader MUST NOT conflict with the existing Go-binary reader: a binary that satisfies BOTH is processed by BOTH, and the output is flat + cross-linked (per Q3 clarification). The SBOM MUST contain exactly ONE file-level component for the binary (emitted by the generic reader, carrying both `mikebom:binary-class = "elf"` and `mikebom:detected-go = true`). BuildInfo modules MUST appear as top-level `pkg:golang/...` components (source-tier), DT_NEEDED linkage evidence MUST appear as top-level `pkg:generic/<soname>` components (analyzed-tier); neither set is nested under the file-level component. Both sets' `evidence.occurrences[]` MUST reference the file-level component's BOM-ref so consumers can trace modules and linkage back to the single physical binary.
- **FR-027**: The generic-binary reader MUST emit `mikebom:binary-stripped = true` for any binary whose format-appropriate symbol/metadata surface is absent: ELF without a symbol-table AND without `.note.package`; Mach-O without `LC_SYMTAB`; PE without `IMAGE_DEBUG_DIRECTORY` entries and without a resource-version block. This signals the consumer that evidence was intrinsically limited.
- **FR-028**: The generic-binary reader MUST NOT emit linkage-evidence components for binaries with zero `DT_NEEDED` / `LC_LOAD_DYLIB` / PE-IMPORT entries (truly static); it still emits the file-level component.
- **FR-028a**: Linkage-evidence components MUST be deduplicated globally by PURL across the scan (per Q5 clarification). When multiple parent binaries reference the same soname / install-name / DLL-name, the SBOM emits exactly one `pkg:generic/<name>` component; every parent binary path MUST be recorded under that single component's `evidence.occurrences[]`. This applies uniformly to ELF, Mach-O, and PE linkage evidence and parallels the PURL-first dedup rule in FR-008.
- **FR-029**: No compositions record is emitted for generic-binary evidence alone — binary analysis is intrinsically heuristic and cannot be declared `aggregate: complete` for any ecosystem. Components from the ELF-note-package path DO count toward their ecosystem's composition (because distro self-identification is authoritative), and MUST participate in the `rpm` / `deb` / `apk` / `alpm` composition tallies the same way the installed-db readers do.

### Key Entities

- **RpmPackageFile**: A `.rpm` artefact on disk. Attributes: path, lead-block-valid (bool), epoch (u32, default 0), name, version, release, arch, license, vendor (header), packager, summary, description, requires (list), provides (list), vendor-source (`header` / `os-release` / `fallback`).
- **ElfBinary**: An ELF file on disk. Attributes: path, class (32/64), data (big-endian/little-endian), elf-type (exec/dyn/rel), machine, stripped (bool), needed (`Vec<String>`), dynamic-linker, note-package (optional structured blob), string-evidence (`Vec<(library, version, offset)>`), linkage-kind, hashes.
- **MachoBinary**: A Mach-O file on disk. Attributes: path, magic, cpu-arch (possibly multi via fat), load-dylib (`Vec<String>`), install-name, stripped (bool), string-evidence, hashes.
- **PeBinary**: A Windows PE file (executable or DLL). Attributes: path, machine, characteristics, import-dlls (`Vec<String>` — merged from IMPORT + Delay-Load IMPORT), stripped (bool), string-evidence, hashes.
- **BdbRpmdb**: A Berkeley-DB rpmdb observed at `/var/lib/rpm/Packages`. Attributes: path, page-size, db-magic, hash-or-btree (classifier), record-count, header-blobs (`Vec<HeaderBlob>`) — each HeaderBlob parsed with the same tag-set as the sqlite reader's row payload.
- **LinkageEvidence**: A per-binary linkage record. Attributes: parent-binary-path, soname-or-install-name-or-dll-name, evidence-kind (`dynamic-linkage`), confidence (`authoritative` for the name-itself, `heuristic` for any name→version-via-embedded-string leap).
- **ElfNotePackage**: A parsed `.note.package` payload. Attributes: type, name, version, architecture, distro, osCpe, raw (original JSON bytes for transparency).
- **EmbeddedVersionMatch**: A hit from the curated version-string scanner. Attributes: parent-binary-path, library (`openssl` / `zlib` / `sqlite` / `curl` / `pcre`), version, offset, confidence (always `heuristic`).

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: A scan of a directory with ≥5 `.rpm` files (real mirror downloads; RHEL 9 / Fedora / Rocky mix) produces one component per `.rpm`, each with a canonical PURL that round-trips through packageurl-python, each with a non-empty license field, each with a non-empty `supplier.name`.
- **SC-002**: For every `.rpm` file whose header declares `REQUIRES:` entries matching another `.rpm`'s `Name:` or `PROVIDES:` in the same scan, the SBOM contains the expected dependency edge. Verified on a fixture with deterministic require/provide pairing.
- **SC-003**: For a stripped ELF binary with ≥3 `DT_NEEDED` entries, the SBOM contains one file-level component plus one linkage-evidence component per `DT_NEEDED` entry, all tagged with the correct evidence-kind.
- **SC-004**: For an ELF binary carrying a `.note.package` section that declares `type: rpm, name: curl, version: 8.2.1, distro: Fedora, arch: x86_64`, the SBOM contains a `pkg:rpm/fedora/curl@8.2.1?arch=x86_64` component — source-tier, authoritative — that passes round-trip conformance against packageurl-python.
- **SC-005**: For a binary with a visible `OpenSSL 3.0.11 19 Sep 2023` marker, the SBOM contains a `pkg:generic/openssl@3.0.11` component tagged `confidence: heuristic`. False-positive rate on a control set of ten Go / Rust binaries with no OpenSSL dependency MUST be zero.
- **SC-006**: A scan of a polyglot artefact directory containing five `.rpm` files AND five ELF binaries completes in under 10 seconds on a modern dev laptop (matches the milestone-003 performance bar) and produces one SBOM containing all deduplicated components.
- **SC-007**: 100% of PURLs emitted by either new reader round-trip through `packageurl-python` (reference-implementation conformance), verified by an integration test per evidence-kind.
- **SC-008**: When a `.rpm` file is malformed (fixture: truncated header), the scan exits cleanly with a single WARN log naming the file and zero components for that file — all other files in the same scan root are processed normally.
- **SC-009**: When a binary is stripped and has zero `DT_NEEDED` entries, the SBOM contains a file-level component with `mikebom:binary-stripped = true` and `mikebom:linkage-kind = "static"` — no fabricated linkage evidence.
- **SC-010**: A scan of a rootfs containing both `/var/lib/rpm/rpmdb.sqlite` (10 packages installed) AND a dropped-in `.rpm` file that is one of those installed packages produces exactly 10 components — deduplication by PURL — with the rpmdb entry as the primary source and the `.rpm`-file reader's evidence merged as a secondary occurrence.
- **SC-011**: For a Windows PE binary whose IMPORT table references `KERNEL32.dll`, `ADVAPI32.dll`, and `libssl-3-x64.dll`, the SBOM contains one file-level component with `mikebom:binary-class = "pe"` and three linkage-evidence components keyed by DLL name; every PURL round-trips through packageurl-python.
- **SC-012**: With `--include-legacy-rpmdb` set, a scan of a RHEL-7 / CentOS-7 / Amazon-Linux-2 rootfs returns a component count within 2% of what `rpm -qa | wc -l` reports on that image, with every component carrying a license from the BDB header. Without the flag, the same scan returns zero rpm components and a single WARN log line (unchanged milestone-003 behaviour).
- **SC-013**: A scan of a mixed artefact directory containing five `.rpm` files, five ELF binaries, three Mach-O binaries, and three PE binaries completes in under 15 seconds on a modern dev laptop and produces one SBOM in which every component carries the correct `mikebom:binary-class` (when binary) and `mikebom:evidence-kind` value.

## Assumptions

- **RPM v3/v4 package-file format is the bar**: legacy RPM v1/v2 formats (rpm 3.x and earlier, pre-2000) are out of scope. If encountered, the reader emits zero components for the file and logs a diagnostic. Ditto for the newer `rpm-ostree` variants not in widespread use.
- **Distro-encoded ELF-note-package payload**: the reader treats the `.note.package` section's JSON contents as the `Packaging Metadata Notes` schema proposed by systemd (roughly 2022) and adopted by Fedora / Debian / Arch (`alpm` variant). Deviations are handled by falling back to `pkg:generic/...` without failing the scan.
- **Curated version-string list is small and fixed**: the v1 set (OpenSSL, zlib, SQLite, curl, PCRE) is encoded in code. Extending the list is a future follow-up and does NOT require a new milestone — each addition is gated by a false-positive test against a known-clean control corpus.
- **Pure-Rust parsers only**: the constitution's "no C" rule (Principle I) applies. The `.rpm`-file parser MUST be a pure-Rust implementation — either a small new crate (e.g. `rpm` / `rpm-rs`) if its transitive dependency tree is constitution-clean, or an in-house parser (~400–700 LOC) if no acceptable crate exists. The `object` crate already in the workspace handles ELF, Mach-O, and PE with no C dependency. The BDB reader is likewise pure-Rust (FR-019).
- **No signature / provenance verification**: RPM PGP-signature verification is a separable concern and belongs in an attestation / provenance milestone, not this one. Same for code-signed Mach-O.
- **No UPX / compressed-payload unpack**: when a binary is packed (UPX signature detected), the scanner annotates and proceeds with what's visible, not unpacks.
- **Windows PE is in scope via IMPORT-table linkage**: PE support is first-class alongside ELF and Mach-O per Q1 resolution. Authenticode code-signing verification remains out of scope (Out of Scope below).
- **Binary analysis is analyzed-tier by default**: linkage evidence is authoritative for *what the binary declares* but heuristic for *what the library actually is at runtime* (DT_NEEDED / LC_LOAD_DYLIB / DLL-name is an identifier, not a PURL). The tier reflects this intrinsic limitation.
- **BDB rpmdb support is opt-in via `--include-legacy-rpmdb`**: default behaviour on pre-RHEL-8 rootfs scans is unchanged from milestone 003 (WARN + zero components). The flag activates the new reader. This conservative gate avoids a silent behaviour change on existing scan targets and lets operators opt into the new output when they're ready.
- **Dedup is PURL-first**: when the `.rpm` reader and rpmdb reader observe the same package, the installed rpmdb entry is authoritative (it's what's actually on the system); the `.rpm`-file evidence merges as a secondary `evidence.occurrences[]` entry.
- **No new deps.dev enrichment for either reader in v1**: both RPM and binary components could benefit from deps.dev (license enrichment, vuln context) later, but MVP is offline-clean. Adding deps.dev enrichment is a stretch goal.

## Dependencies

- Milestone 003 (Go / RPM / Maven / Cargo / Gem) infrastructure: `rpm_vendor_from_id`, `PackageDbEntry` fields, scan_fs edge-resolver, `sbom_tier` ladder, `mikebom:source-type` / `mikebom:evidence-kind` / `mikebom:dev-dependency` properties — all reused as-is.
- The `object` crate (already in the workspace) — covers ELF, Mach-O, AND PE parsing without adding C code.
- `packageurl-python` reference implementation — required at test time for round-trip conformance (already used in milestones 002 / 003).
- A pure-Rust RPM-file parser — either a vetted existing crate (candidate: the `rpm` crate; constitution-vetted during planning) or an in-house implementation. Decision made during planning; either is acceptable.
- A pure-Rust Berkeley-DB page reader (new dependency for US4) — either a vetted existing crate whose transitive tree passes the Principle-I "no C" check, or an in-house subset reader covering Hash + BTree page layouts and the HeaderBlob record format. Milestone-003's pure-Rust rpmdb-sqlite reader sets the precedent; the BDB reader is a parallel effort with a similar shape.

## Out of Scope

- **RPM PGP signature verification** (`rpm --checksig` equivalent): belongs in a provenance / attestation milestone.
- **Mach-O code-signing verification** (`codesign -v` equivalent): ditto.
- **PE Authenticode verification** (signing chain, timestamp, CRL / OCSP checks): ditto. The certificate table is skipped during parsing.
- **.NET / CLI assembly metadata for PE binaries**: only the unmanaged IMPORT table is parsed. Deeper CLI introspection (assembly-reference tables, strong-name signatures) is a separate future effort.
- **SRPM source-tarball walking**: the `.rpm` reader records `arch=src`; it does not unpack the source payload.
- **UPX / ASProtect / packed-binary unpack**: annotate + proceed, don't unpack.
- **BDB master-password / transaction-log handling**: the legacy-rpmdb reader parses standalone `Packages` files only; encrypted or transaction-log-locked databases are out of scope.
- **Firmware / UEFI images**: explicit non-goal; binary analysis in this milestone means executable binaries, not flashable firmware blobs.
- **Kernel modules** (`.ko`): ELF technically, but the linkage model is different; out of scope.
- **Container image layer tarballs**: scanning `.tar.gz` layer blobs directly (rather than extracted rootfs) is a separate docker-image capability and already separately tracked.
- **deps.dev enrichment for new readers in v1**: both new readers could leverage deps.dev later for license / vuln enrichment; not a milestone-004 requirement.
- **Extending the curated embedded-version-string library list beyond the v1 five (OpenSSL, zlib, SQLite, curl, PCRE)**: each addition is incremental, not a milestone gate.
- **Fuzz-testing harness for `.rpm` header parsing and ELF `.note.package` parsing**: defense-in-depth baseline (FR-007) is the milestone-004 bar; continuous fuzzing is a post-milestone follow-up — same handling as milestone 003's parser-fuzz TODO.
