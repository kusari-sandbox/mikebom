# Implementation Plan: RPM Package-File Scanning & Generic Binary SBOMs

**Branch**: `004-rpm-binary-sboms` | **Date**: 2026-04-18 | **Spec**: [`spec.md`](./spec.md)
**Input**: Feature specification from `/specs/004-rpm-binary-sboms/spec.md`

## Summary

Extend `mikebom sbom scan` with two artefact-level readers and one opt-in legacy path:

1. **Standalone `.rpm` file reader (US1, P1)** — parse RPM v3/v4 package headers (lead + main header blob) from `.rpm` files on disk, emit one `pkg:rpm/<vendor>/<name>@<epoch>:<version>-<release>?arch=<arch>` component per file with license / supplier / requires / provides populated from header tags. Vendor segment derived via priority `Vendor:` header → os-release `ID` → `rpm` fallback.
2. **Generic (non-Go) binary reader (US2, P2)** — cross-platform ELF / Mach-O / PE parser that emits (a) a file-level component per binary, (b) linkage-evidence components for every `DT_NEEDED` / `LC_LOAD_DYLIB` / PE IMPORT / PE Delay-Load IMPORT entry, (c) source-tier components from ELF `.note.package` distro self-identification, and (d) heuristic `pkg:generic/<lib>@<ver>` components from the bounded curated embedded-version-string scanner (OpenSSL, zlib, SQLite, curl, PCRE) running only inside format-appropriate read-only string sections. All linkage evidence is deduplicated globally by PURL with per-parent-binary occurrences merged.
3. **Opt-in legacy Berkeley-DB rpmdb reader (US4, P3)** — new `--include-legacy-rpmdb` flag activates a pure-Rust BDB Hash/BTree-page + HeaderBlob subset reader for `/var/lib/rpm/Packages` on pre-RHEL-8 / CentOS-7 / Amazon-Linux-2 rootfs. Unflagged default behaviour is unchanged from milestone 003 (WARN + zero components).

Supporting work bundled into this milestone: canonicalize both rpmdb paths' `mikebom:evidence-kind` values to `rpmdb-sqlite` / `rpmdb-bdb` (one-line update to milestone-003's `rpm.rs` at serialization time); a new `mikebom:evidence-kind` property appears on every rpm, linkage, ELF-note, and embedded-string component so consumers can filter by source-of-truth. No `mikebom-common` schema changes; only additive `PackageDbEntry` field extensions + new CycloneDX properties, all backward-compatible.

## Technical Context

**Language/Version**: Rust stable, same workspace compiler as milestones 001–003. No new nightly-only features. `mikebom-ebpf` is untouched.

**Primary Dependencies (deltas)**:

- **`object` crate (bump features)** — currently `0.36 { features = ["read", "std", "elf", "macho"] }`; add `"pe"` + `"coff"` features to cover PE IMPORT / Delay-Load IMPORT directories. `object`'s PE parser is pure-Rust; no `pelite` / `goblin` needed.
- **`rpm` crate (NEW candidate)** — pure-Rust RPM v3/v4 header reader. Research R1 decides between (a) adopting `rpm` (latest stable, ~2.0.x) if its transitive tree is constitution-clean or (b) hand-rolling a ~500 LOC subset reader (lead block + header-index + header-blob tag enumeration — same shape as the sqlite reader's row payload, which is itself a HeaderBlob).
- **Pure-Rust Berkeley-DB page reader (NEW candidate)** — research R2 evaluates the ecosystem. Candidates: `berkeleydb` (crates.io, status TBD), `bdb` (TBD), or an in-house ~700 LOC subset reader covering Hash + BTree page layouts and HeaderBlob records. Precedent: milestone 003's `rpmdb_sqlite/` subdirectory.
- **No new ELF `.note.package` crate** — hand-parse the 32-byte note header + JSON payload (~40 LOC); section lookup already available via `object::Object`.
- **No new curated-string matcher crate** — a 5-entry regex or byte-pattern table in a new `binary/version_strings.rs` module (~100 LOC).
- **`packageurl`** — unchanged; canonical PURLs round-trip through `packageurl-python` per FR-002.

**Storage**: N/A — all state is in-process for the duration of a single scan. Mirrors milestones 002 / 003.

**Testing**: `cargo test --workspace` for unit + integration. Fixtures strategy inherited from milestones 002 / 003:

- **Unit tests** live inline in each new module (`scan_fs/package_db/rpm_file.rs`, `scan_fs/package_db/rpmdb_bdb/`, `scan_fs/binary/`) with synthetic inputs constructed via `tempfile::tempdir()` helpers.
- **Integration tests** under `mikebom-cli/tests/scan_<area>.rs` shell out to `CARGO_BIN_EXE_mikebom` and parse resulting JSON SBOMs.
- **Real fixtures** under `tests/fixtures/rpm-files/`, `tests/fixtures/binaries/elf/`, `tests/fixtures/binaries/macho/`, `tests/fixtures/binaries/pe/`, and `tests/fixtures/bdb-rpmdb/`. Policy: small real `.rpm` / compiled binaries committed directly (sub-1 MB each); larger blobs regenerated by a one-shot helper script documented in `quickstart.md`.
- **PE fixtures** produced by cross-compiling a trivial Rust crate with `x86_64-pc-windows-gnu` (no Wine needed to run — we parse, we don't execute); Mach-O fixtures cross-compiled with `aarch64-apple-darwin` or picked up from macOS CI runners.

**Target Platform**: Linux (primary), macOS (CI). `mikebom sbom scan` is platform-agnostic; binary parsers run against any of ELF / Mach-O / PE regardless of host. `mikebom-ebpf` (Mode 1 trace) is unaffected.

**Project Type**: CLI tool — additions to the existing `mikebom sbom scan` subcommand, no new top-level subcommands.

**Performance Goals** (carried from spec + dev-laptop budgets):

- SC-006: five `.rpm` files + five ELFs in one scan under 10 s (milestone-003 bar).
- SC-013: five `.rpm` + 5 ELF + 3 Mach-O + 3 PE in one scan under 15 s.
- SC-012 / FR-019b: BDB rpmdb with ≥1000 package rows parses in ≤2 s (iteration budget mirrors the sqlite reader).
- Per-`.rpm`-file parse budget: ≤50 ms for a 1 MB `.rpm` (header-only parse; payload is ignored).
- Per-binary parse budget: ≤100 ms for a 10 MB ELF (section-restricted string scan + IMPORT-table walk).

**Constraints**:

- **Pure Rust, zero C** — Principle I binds every new parser. Research R1 (`.rpm` crate), R2 (BDB reader), and R3 (PE via `object`) each confirm clean transitive trees or go in-house.
- **No `.unwrap()` in production** — Principle IV. `#![deny(clippy::unwrap_used)]` is already set at crate root (`main.rs:7`); new modules inherit it; tests opt back in via `#[cfg_attr(test, allow(clippy::unwrap_used))]`.
- **Offline-clean** — FR-006. Both readers and the legacy BDB reader emit complete output without network access. No new deps.dev enrichment in v1.
- **Hardened untrusted parsing** — FR-007. Mirrors milestone 003's baseline: per-file size caps, magic-byte validation, bounded iteration counts with wall-clock budgets, fail-graceful with single WARN + zero components.
- **Backward-compatible output** — existing SBOM consumers continue to parse the output. New `mikebom:evidence-kind` property is additive; canonicalizing the sqlite reader's value is a thin retrofit (it emits NO evidence-kind today, so there's nothing to break — we're introducing the property universally).

**Scale/Scope**: ~1800–2600 LOC of new Rust split across:

- `mikebom-cli/src/scan_fs/package_db/rpm_file.rs` — ~300–500 LOC (or ~150 LOC if `rpm` crate adopted).
- `mikebom-cli/src/scan_fs/package_db/rpmdb_bdb/` — ~600–900 LOC new submodule.
- `mikebom-cli/src/scan_fs/binary/` — ~500–800 LOC new submodule (ELF DT_NEEDED + `.note.package` + Mach-O LC_LOAD_DYLIB + PE IMPORT + version-string scanner + format dispatcher).
- `mikebom-cli/src/scan_fs/package_db/rpm.rs` — small retrofit (~30 LOC) to emit `mikebom:evidence-kind = "rpmdb-sqlite"`.
- `mikebom-cli/src/cli/scan_cmd.rs`, `src/main.rs`, `scan_fs/mod.rs::scan_path` signature — flag wiring for `--include-legacy-rpmdb` (~20 LOC).
- `mikebom-cli/src/generate/cyclonedx/builder.rs` — ~50 LOC to serialize the new properties.
- ~1200 LOC of tests + fixtures.

Roughly 0.9× the milestone-003 surface area.
