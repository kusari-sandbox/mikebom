# Feature Specification: SPDX Output Support (2.3 with groundwork for 3+)

**Feature Branch**: `010-spdx-output-support`
**Created**: 2026-04-23
**Status**: Draft
**Input**: User description: "I want to now support SPDX. I want to start with SPDX 2.3 but also lay the groundwork for SPDX 3+. In addition, I want to ensure that at every step we ensure that if we are putting something into CycloneDX we look at the best place to put it in SPDX as well. We should also document where some data goes into in each spec."

## Clarifications

### Session 2026-04-23

- Q: SPDX 3+ groundwork depth — documentation only, documentation + interface shaping, or also a minimal working stub? → A: Option C — documentation + interface shaping + a minimal-but-valid SPDX 3 emitter for at least one currently-supported ecosystem, gated behind an explicit opt-in. The stub is shipped in this milestone alongside SPDX 2.3, not deferred.
- Q: Fallback policy for mikebom data with no SPDX 2.3 home? → A: Option A — SPDX `annotations[]` with mikebom-namespaced JSON-encoded comments is the default fallback for `mikebom:*` properties, evidence (identity + occurrences), and compositions. CycloneDX `vulnerabilities[]` (VEX) is the documented exception: it is emitted as a separate OpenVEX JSON sidecar file alongside the SPDX file and referenced from the SPDX document.
- Q: Concurrent dual-format emission UX — one scan, multiple formats from a single invocation? → A: Option A — a single `mikebom sbom scan` invocation can produce CycloneDX and SPDX (and the OpenVEX sidecar) from one scan in v1. The format selector accepts a list (e.g., `--format cyclonedx-json,spdx-2.3-json`). When multiple formats are requested, each is written under its own default filename unless the user overrides; rescanning is never required to obtain the second format.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Produce a valid SPDX 2.3 SBOM from a scan (Priority: P1)

A user who today runs `mikebom sbom scan` against a filesystem or container image and receives a CycloneDX SBOM wants to produce an **equivalent SPDX 2.3 JSON SBOM** from the same scan, covering the same set of components across all currently supported ecosystems (apk, cargo, deb, gem, golang, maven, npm, pip, rpm). The resulting file must be accepted as valid by the official SPDX validator and by common SPDX consumers, and must carry the same core identity facts (PURL, name, version, supplier where known, checksums, declared and concluded licenses, dependency relationships) that the CycloneDX output carries for the same scan.

**Why this priority**: Without this, mikebom only serves the CycloneDX half of the SBOM-consumer market. Users standardized on SPDX (many US-federal procurement paths, Linux Foundation tooling, many enterprise SCA pipelines) cannot adopt mikebom at all today. Baseline format parity is the minimum viable delivery.

**Independent Test**: Run `mikebom sbom scan` against a fixture project (e.g., a small multi-ecosystem test tree such as the existing integration-test fixtures for npm + maven + deb) requesting SPDX output; assert the produced file parses cleanly with an off-the-shelf SPDX 2.3 validator, passes `sbomqs` scoring at or above the CycloneDX baseline for equivalent fields, and contains one SPDX Package per component that also appears in the CycloneDX output for the same scan, with matching PURL / version / checksum values.

**Acceptance Scenarios**:

1. **Given** a filesystem tree containing a populated `node_modules/` and `package-lock.json`, **When** the user requests an SPDX 2.3 JSON SBOM, **Then** every npm package present in the equivalent CycloneDX output appears as an SPDX Package with matching `name`, `versionInfo`, `externalRefs[PURL]`, and `checksums`, and the document validates against the SPDX 2.3 JSON schema.
2. **Given** a Debian container image, **When** the user requests an SPDX 2.3 JSON SBOM, **Then** every `deb`-ecosystem component from the CycloneDX output appears as an SPDX Package with the same PURL (including `arch` and `distro` qualifiers) and the same SHA-256 checksums, and the document's `documentDescribes` / root-level relationship points to the image/filesystem root.
3. **Given** a scan that produces components with both `declared` and `concluded` licenses in CycloneDX, **When** the user requests SPDX 2.3 output, **Then** those values land in SPDX's native `licenseDeclared` and `licenseConcluded` fields on the corresponding Packages, preserving the SPDX license expression string.
4. **Given** the same scan run twice with the same inputs, **When** each run produces SPDX 2.3 output, **Then** the resulting documents are byte-for-byte identical except for fields the SPDX spec requires to vary per document (e.g., `SPDXID` of the document itself and `created` timestamp), matching mikebom's existing determinism posture for CycloneDX.
5. **Given** a single `mikebom sbom scan` invocation requesting both CycloneDX and SPDX 2.3 output, **When** the scan completes, **Then** both files are produced under their respective default filenames (`mikebom.cdx.json` and `mikebom.spdx.json`) from a single pass over the target — the target is not rescanned, deep-hash work is not duplicated — and the contents of each file are bit-identical to what would be produced by invoking the CLI separately for each format.

### User Story 2 - Dual-format data-placement guarantee and documentation (Priority: P2)

A user, auditor, or downstream tool author needs to trust that any fact mikebom emits in one format is **also present in the other format, in the most semantically-correct location that format offers**. For every data element mikebom currently writes into CycloneDX — PURLs, checksums, declared/concluded licenses, supplier/author, external references, relationships, dependency edges, evidence (identity + occurrences), custom `mikebom:*` properties, compositions/completeness markers, and embedded VEX — there is a documented target location in SPDX 2.3 (or a documented, justified reason why it is omitted), and a documented forward-path for SPDX 3+. This documentation is shipped with the project so users can reconcile SBOMs produced in either format for the same scan.

**Why this priority**: This is the core of the user's request ("at every step we ensure that if we are putting something into CycloneDX we look at the best place to put it in SPDX as well"). Without it, format output becomes divergent over time and SPDX consumers silently lose mikebom's value-add (provenance, confidence, evidence). Making the mapping explicit and testable prevents drift.

**Independent Test**: Open the committed dual-format data-placement map (e.g., `docs/reference/sbom-format-mapping.md`). For every row, run an SPDX scan and a CycloneDX scan against a fixture that exercises that field and confirm the documented target location actually holds the expected value (or that the documented "omitted in SPDX 2.3" reason is surfaced in the SBOM in the agreed fallback location).

**Acceptance Scenarios**:

1. **Given** the shipped CycloneDX-to-SPDX data-placement map, **When** a reviewer walks each CycloneDX field that mikebom emits today (listed in `docs/design-notes.md` and observable in existing fixture outputs), **Then** each row specifies: the CycloneDX location, the SPDX 2.3 location (field or relationship, or "omitted — reason"), the SPDX 3 location, and a one-line justification.
2. **Given** a scan that emits a custom mikebom property (e.g., `mikebom:evidence-kind`, `mikebom:shade-relocation`, `mikebom:binary-class`), **When** SPDX 2.3 output is produced, **Then** the value lands in the location named in the data-placement map (e.g., SPDX `annotations[]` with a mikebom-namespaced comment, or an agreed `externalRefs` entry), consistently across ecosystems.
3. **Given** a scan that emits CycloneDX `evidence.identity` and `evidence.occurrences`, **When** SPDX 2.3 output is produced, **Then** the equivalent facts appear in the SPDX location specified by the map, and the SPDX output does not silently drop them without the user being able to see why from the documentation.
4. **Given** a scan that emits CycloneDX `vulnerabilities[]` (VEX), **When** SPDX 2.3 output is produced, **Then** an OpenVEX JSON sidecar file (`mikebom.openvex.json` by default) is written next to the SPDX file carrying the same VEX statements, and the SPDX document references the sidecar (via `externalDocumentRefs` or a document-level annotation) so a consumer reading only the SPDX file can discover it. **Given** a scan that emits no VEX statements, no sidecar file is created.

### User Story 3 - SPDX 3+ groundwork plus minimal stub emitter (Priority: P3)

A maintainer (and an early adopter willing to opt in) wants two things from SPDX 3 in this milestone: (a) the structural insurance that adding full SPDX 3 emission later is incremental, not a rewrite — meaning the internal model is format-neutral, the data-placement map already carries the SPDX 3 column, and the CLI format-dispatch layer is open for extension; and (b) a working **minimal-but-valid SPDX 3 stub emitter** for at least one currently-supported ecosystem, gated behind an explicit opt-in (e.g., an experimental format identifier), so that the structural design is exercised by real code today, not just on paper.

**Why this priority**: SPDX 3 is a fundamentally different model (element graph, profiles, namespace-versioned vocabularies) rather than a minor evolution of 2.3. Documenting alone (option A) leaves the structural risk theoretical; shipping a stub (option C) forces us to actually exercise the format-neutral model and the dispatch layer with a second SPDX target now, while keeping the surface narrow (one ecosystem, opt-in). Still P3 because it is not the format that day-one users will publish — SPDX 2.3 (P1) and the cross-format mapping (P2) deliver immediate value; SPDX 3 stub is forward-looking and explicitly experimental.

**Independent Test**: With the experimental SPDX 3 opt-in selected, run `mikebom sbom scan` against a fixture for the chosen ecosystem and assert: (i) the produced SPDX 3 document validates against the targeted SPDX 3.x JSON schema, (ii) it covers every component the CycloneDX output covers for the same ecosystem fixture (parity on PURL, name, version, checksums), (iii) producing it required no changes to scan, resolution, or the CycloneDX serializer relative to the SPDX-2.3-only baseline. Separately, a maintainer reading the internal format-serializer interface and the data-placement map can describe — without reading further implementation code — how to extend the stub to additional ecosystems and SPDX 3 profiles in a follow-up milestone.

**Acceptance Scenarios**:

1. **Given** the internal model types used by the SPDX 2.3 and SPDX 3 stub serializers, **When** a maintainer inspects them, **Then** those types are the same neutral scan-result types used by CycloneDX, not SPDX-2.3-specific or SPDX-3-specific structs, so additional SPDX 3 profile coverage can be added alongside without coupling to other formats.
2. **Given** the data-placement map, **When** a reviewer reads any row, **Then** it carries columns for CycloneDX, SPDX 2.3, and SPDX 3, with every SPDX 3 entry populated (either a concrete target location/profile, or "defer until SPDX 3 profile X stabilizes" with a cited reason).
3. **Given** the CLI format-dispatch code, **When** a maintainer adds another format value or extends the SPDX 3 stub to a new ecosystem, **Then** they only add or extend a serializer implementation and register it; they do not modify scanning, resolution, or other serializers.
4. **Given** a fixture for the chosen ecosystem and the experimental SPDX 3 opt-in selected, **When** the user runs `mikebom sbom scan`, **Then** the produced SPDX 3 JSON document validates against the targeted SPDX 3.x JSON schema and contains the same components (matched by PURL) as the CycloneDX output for the same scan.
5. **Given** the experimental SPDX 3 opt-in is **not** selected, **When** the user runs `mikebom sbom scan`, **Then** behavior is identical to a build without the SPDX 3 stub at all — no SPDX 3 file is produced, no SPDX 3 dependencies are loaded, and no warnings reference SPDX 3.

### Edge Cases

- **Nested components (shade-relocation ancestor, shade-jar children)**: CycloneDX nests components inside other components. SPDX 2.3 has a flat Package list with explicit `CONTAINS` / `CONTAINED_BY` relationships. The SPDX output must preserve the same containment facts via relationships; it must not silently flatten the hierarchy into an undifferentiated list.
- **Multiple roots / no root**: Scans of multi-project filesystem trees may produce multiple "root" items or none. SPDX requires a `documentDescribes` target. The output must pick a deterministic root (synthesizing one if needed) and document the choice.
- **Missing supplier/originator**: Many ecosystems (e.g., vendored jars, scanned deb packages without DEBIAN/control metadata) have no supplier. SPDX 2.3 requires either a value or the literal `NOASSERTION`. The output must emit `NOASSERTION` rather than omitting the field or making up a value.
- **License data mikebom cannot parse as an SPDX expression**: CycloneDX accepts both single SPDX IDs and free-text expressions. SPDX 2.3 requires `licenseConcluded` / `licenseDeclared` to be either a valid SPDX expression, `NOASSERTION`, or `NONE`. Invalid expressions must be emitted as `NOASSERTION` with the raw text preserved in the agreed fallback location from the data-placement map, not silently dropped.
- **VEX / vulnerability data**: CycloneDX carries these natively; SPDX 2.3 does not. Per FR-016a, when VEX statements exist for a scan, an OpenVEX JSON sidecar (`mikebom.openvex.json` by default) is written next to the SPDX file and referenced from it; when no VEX statements exist, no sidecar is written. The sidecar's existence and discovery path are visible to a consumer reading only the SPDX file.
- **Compositions / ecosystem completeness markers**: CycloneDX uses the `compositions` array to mark "complete ecosystem coverage" claims. SPDX 2.3 has no direct equivalent. Per FR-016, these are emitted as document-level SPDX `annotations[]` with a mikebom-namespaced JSON payload, recorded in the data-placement map.
- **Scans producing zero components**: An empty scan must still yield a valid SPDX 2.3 document (a document with zero Packages and a `documentDescribes` pointing to the scan root), not a parse error in downstream tools.
- **Very large component counts** (tens of thousands of Packages, e.g., a full OS image): Output must remain valid and must not exhaust memory beyond the existing CycloneDX ceiling for the same scan.

## Requirements *(mandatory)*

### Functional Requirements

**Format selection and output**

- **FR-001**: The `mikebom sbom scan` command MUST accept a user-specified request for SPDX 2.3 JSON output as a peer of the existing CycloneDX JSON output.
- **FR-002**: Selecting SPDX output MUST NOT change, skip, or reorder any step of the scan or resolution pipeline; SPDX emission is strictly a serialization choice over the same internal scan result.
- **FR-003**: When SPDX output is selected, the default output filename MUST follow the ecosystem convention `mikebom.spdx.json` (analogous to the existing `mikebom.cdx.json` default).
- **FR-004**: A single `mikebom sbom scan` invocation MUST be able to emit any non-empty subset of the supported output formats (e.g., `cyclonedx-json` alone, `spdx-2.3-json` alone, both together, or both together with the experimental `spdx-3-json` stub from FR-019a) from a single scan, without rescanning the target. The format selector MUST accept a list (e.g., comma-separated values, repeated flag occurrences, or equivalent). When the same format is requested more than once in a single invocation, the duplicate MUST be ignored rather than treated as an error.
- **FR-004a**: When multiple formats are emitted from a single invocation, each format's output MUST be written under its own default filename (`mikebom.cdx.json`, `mikebom.spdx.json`, etc., with the OpenVEX sidecar from FR-016a alongside the SPDX file when applicable) unless the user overrides paths. The user MUST be able to override per-format output paths so files can be directed to different locations. Output filename collisions across formats MUST be impossible by default (defaults are distinct per format) and MUST be reported as a clear user-facing error if produced by user-supplied overrides.
- **FR-004b**: Default behavior when no format is specified MUST remain unchanged from the pre-milestone baseline: a single CycloneDX JSON file at the existing default path. SPDX is opt-in; users do not silently start receiving SPDX output after this milestone is shipped.

**SPDX 2.3 document structure**

- **FR-005**: Produced SPDX 2.3 JSON documents MUST validate against the official SPDX 2.3 JSON schema without warnings or errors that a user would not also see from a reputable reference SBOM of similar shape.
- **FR-006**: Every component mikebom emits in CycloneDX for a given scan MUST appear as exactly one SPDX Package in the SPDX output for the same scan, with stable `SPDXID` values derived deterministically from the component's identity (so re-running the scan produces the same IDs).
- **FR-007**: Each SPDX Package MUST carry the component's PURL in `externalRefs[]` using the SPDX-defined `PACKAGE-MANAGER` / `purl` reference type, preserving all PURL qualifiers (arch, distro, epoch, etc.) unchanged from the CycloneDX output.
- **FR-008**: Each SPDX Package MUST carry every checksum mikebom emits for that component in CycloneDX, using SPDX's native `checksums[]` array with the matching algorithm identifiers (`SHA256`, `SHA512`, `SHA1`, `MD5`, etc.).
- **FR-009**: Declared license data MUST land in SPDX `licenseDeclared`; concluded license data MUST land in SPDX `licenseConcluded`. Where mikebom has no value for one of these fields, the SPDX output MUST emit `NOASSERTION`, never a fabricated value.
- **FR-010**: Dependency edges that appear in CycloneDX `dependencies[]` MUST be emitted as SPDX Relationships with appropriate relationship types (`DEPENDS_ON` for runtime, `DEV_DEPENDENCY_OF` where mikebom knows dev scope, etc.), such that the graph of relationships in the SPDX output is isomorphic to the CycloneDX dependency graph for the same scan.
- **FR-011**: Containment relationships that CycloneDX expresses via nested components (e.g., shade-jar children, image → filesystem layers) MUST be emitted as explicit SPDX `CONTAINS` / `CONTAINED_BY` relationships between flat Packages.
- **FR-012**: The SPDX document MUST carry a `documentDescribes` entry identifying the scan root (filesystem path or image reference). If a scan has no natural single root, a synthetic root Package representing "this scan" MUST be emitted and described, with the choice being deterministic across runs.

**Dual-format data-placement mapping**

- **FR-013**: The project MUST ship a committed data-placement map (expected location: `docs/reference/sbom-format-mapping.md` or equivalent under `docs/`) that, for every data element mikebom emits today in CycloneDX, specifies: (a) the CycloneDX location, (b) the SPDX 2.3 location (field path, relationship type, or "omitted — reason"), (c) the SPDX 3 location (field path, profile, or "defer — reason"), and (d) a one-line justification for the placement choice.
- **FR-014**: The data-placement map MUST cover at minimum: PURL, name, version, supplier/originator, checksums, declared license, concluded license, dependency edges, nested/containment relationships, external references (homepage, VCS, deps.dev), evidence (CycloneDX `evidence.identity` + `evidence.occurrences`), compositions/completeness markers, CycloneDX `vulnerabilities[]` (VEX), and every `mikebom:*` property currently emitted (source-type, source-connection-ids, deps-dev-match, evidence-kind, sbom-tier, dev-dependency, co-owned-by, shade-relocation, npm-role, binary-class, binary-stripped, linkage-kind, buildinfo-status, detected-go, binary-packed, confidence, raw-version, source-files, cpe-candidates, requirement-range, generation-context, os-release-missing-fields, trace-integrity-* metadata).
- **FR-015**: For every data-placement map row whose SPDX 2.3 column is not "omitted", the SPDX output MUST actually populate that location when the corresponding CycloneDX output populates its location, for the same scan.
- **FR-016**: For mikebom-specific data that has no direct SPDX 2.3 field — including every `mikebom:*` property, CycloneDX `evidence.identity` and `evidence.occurrences`, and `compositions` / completeness markers — the SPDX 2.3 output MUST preserve the data losslessly as SPDX `annotations[]` entries on the document or the relevant Package, with a mikebom-namespaced JSON-encoded comment value (so consumers that ignore annotations see a clean SPDX, while consumers that read them recover full mikebom fidelity). The annotation namespace (e.g., `mikebom:<field>`) MUST be consistent across ecosystems and MUST be documented in the data-placement map. Silent dropping is prohibited.
- **FR-016a**: CycloneDX `vulnerabilities[]` (VEX) is the documented exception to FR-016. When mikebom would emit VEX into CycloneDX for a given scan, the SPDX 2.3 output MUST be accompanied by a separate **OpenVEX JSON sidecar file** (default name `mikebom.openvex.json`, written next to the SPDX file) carrying the same VEX statements. The SPDX document MUST reference the sidecar via an SPDX `externalDocumentRefs` entry (or, if that is not semantically appropriate, a document-level annotation that names the sidecar's relative path and `sha256`), so that a consumer reading only the SPDX file can discover the sidecar. When a scan produces no VEX statements, no sidecar is written.

**SPDX 3+ groundwork**

- **FR-017**: Internal types consumed by the SPDX 2.3 serializer and the SPDX 3 stub serializer MUST be the same format-neutral scan-result types already used by the CycloneDX serializer; no SPDX-2.3-specific or SPDX-3-specific struct may leak upward into scan or resolution code.
- **FR-018**: The data-placement map MUST include a populated SPDX 3 column for every row. Entries are either a concrete target location (field path, profile, relationship type) or "defer until SPDX 3 profile X stabilizes" with a cited reason. "Unknown" is not an acceptable entry.
- **FR-019**: The CLI format-dispatch code MUST be structured so that adding a new format value, or extending the SPDX 3 stub to additional ecosystems and profiles in a future milestone, is a matter of adding/extending a serializer implementation and registering it; it MUST NOT require modifying scanning, resolution, the CycloneDX serializer, or the SPDX 2.3 serializer.
- **FR-019a**: This milestone MUST ship a minimal-but-valid SPDX 3 JSON stub emitter covering at least one currently-supported ecosystem, gated behind an explicit experimental opt-in (e.g., a distinct format identifier such as `spdx-3-json-experimental` or equivalent). The stub MUST validate against the targeted SPDX 3.x JSON schema and MUST cover, for the chosen ecosystem, the same components the CycloneDX output covers (parity on PURL, name, version, checksums). The opt-in MUST be off by default; when not selected, behavior MUST be byte-identical to a build without the stub.
- **FR-019b**: The experimental opt-in MUST be visibly labeled as such in user-facing surfaces (CLI `--help` text, format identifier name, output-document creator-tool comment in the produced SPDX 3 file) so that users do not mistake the stub for production-grade SPDX 3 emission. Coverage limitations (which ecosystem(s), which SPDX 3 profiles, which fields in the data-placement map are not yet honored on the SPDX 3 side) MUST be documented in the same place as the data-placement map.

**Determinism, validation, parity**

- **FR-020**: SPDX output MUST be deterministic across repeated runs with identical inputs, to the same standard as CycloneDX output today: identical bytes except for the SPDX-mandated document-level identifiers and timestamps.
- **FR-021**: The project MUST ship automated tests that, for each currently-supported ecosystem (apk, cargo, deb, gem, golang, maven, npm, pip, rpm), validate SPDX 2.3 output from a fixture against the SPDX 2.3 JSON schema and assert parity with CycloneDX output on the core identity fields per the data-placement map.
- **FR-022**: SPDX output MUST NOT regress any existing CycloneDX output: CycloneDX files produced after this milestone MUST be byte-identical to those produced before it for the same scan (the feature is additive).

### Key Entities

- **SBOM Output Format**: A selectable format target for emitting the scan result. Attributes: identifier (e.g., `cyclonedx-json`, `spdx-2.3-json`, forward-looking `spdx-3-json`), default filename, whether it is "stable" or "groundwork-only". The existing CycloneDX format is one instance; this milestone adds SPDX 2.3 as a second and reserves a slot for SPDX 3.
- **Dual-Format Data-Placement Map**: A committed, reviewable document that rows every data element mikebom emits with a column per format (CycloneDX, SPDX 2.3, SPDX 3). Serves as both contract (the SPDX serializer must honor it) and documentation (consumers can reconcile SBOMs across formats). Owned by this milestone; expected to be updated in lockstep any time a new `mikebom:*` property or CycloneDX field is added in future milestones.
- **SPDX Document**: The envelope for SPDX output. Carries document-level metadata (creation info, document namespace, describes relationships) and contains Packages and Relationships.
- **SPDX Package**: The unit mikebom emits per scan component — the SPDX equivalent of a CycloneDX component. Carries name, version, supplier/originator, checksums, declared and concluded licenses, external references (including PURL), and annotations/comments for mikebom-specific fallback data.
- **SPDX Relationship**: An edge between two SPDX elements. Used to express dependency edges (`DEPENDS_ON`, `DEV_DEPENDENCY_OF`), containment (`CONTAINS`, `CONTAINED_BY`), and the document-describes root pointer.
- **OpenVEX Sidecar**: A separate OpenVEX JSON file (default name `mikebom.openvex.json`) emitted next to the SPDX file when a scan produces VEX statements. Carries the same VEX content that CycloneDX would include in its `vulnerabilities[]` array. Discoverable from the SPDX document via `externalDocumentRefs` or a document-level annotation. Only created when VEX statements are present.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: A user running the same scan can produce a valid SPDX 2.3 JSON SBOM with complete ecosystem coverage for all 9 currently-supported ecosystems, achieving at least parity with the CycloneDX output's `sbomqs` score on fields both formats express natively (name, version, supplier, checksums, license, dependency relationships, external references).
- **SC-002**: SPDX 2.3 output produced by mikebom validates clean (zero errors, and only warnings that the SPDX-published reference example `SPDXJSONExample-v2.3.spdx.json` from the SPDX 2.3 spec repository at `https://github.com/spdx/spdx-spec/tree/development/v2.3/examples/` also produces under the same validator) against the SPDX 2.3 JSON schema using a maintained pure-Rust JSON Schema validator. CI runs this validation per ecosystem (FR-021). The reference example bytes are vendored under `mikebom-cli/tests/fixtures/reference/SPDXJSONExample-v2.3.spdx.json` so the warning baseline is reproducible without network access.
- **SC-003**: For every ecosystem, every component that appears in the CycloneDX output for a given scan appears as exactly one SPDX Package in the SPDX output for the same scan, with matching PURL and checksum values — verified by an automated cross-format parity test.
- **SC-004**: The committed dual-format data-placement map covers 100% of the data elements mikebom emits in CycloneDX today (including all `mikebom:*` properties, evidence, compositions, external references, and VEX), with a non-empty entry for each of the three format columns (CycloneDX, SPDX 2.3, SPDX 3) — verified by a documentation-completeness check in CI against a checklist derived from the existing CycloneDX output.
- **SC-005**: A maintainer unfamiliar with this milestone can read the serializer interface, the data-placement map, and the SPDX 3 stub and describe, in under 15 minutes and without reading further implementation code, exactly what it would take to extend the SPDX 3 stub to additional ecosystems and profiles in a future milestone — with the answer being "extend the existing SPDX 3 serializer using the format-neutral model and the SPDX 3 column of the map, no changes to scan, resolution, the CycloneDX serializer, or the SPDX 2.3 serializer."
- **SC-006**: Existing CycloneDX output for any scan is byte-identical before and after this milestone — verified by a regression test against a pinned set of fixture outputs.
- **SC-007**: SPDX 2.3 output is deterministic across repeated runs on identical inputs, to the same standard as CycloneDX — verified by a reproducibility test that diffs two sequential runs and asserts identity except for document-mandated time/id fields.
- **SC-008**: With the SPDX 3 experimental opt-in selected, the produced SPDX 3 JSON document for the chosen ecosystem fixture validates clean against the targeted SPDX 3.x JSON schema and matches CycloneDX component coverage by PURL for that ecosystem — verified by an automated test. With the opt-in not selected, scan output is byte-identical to a build without the SPDX 3 stub — verified by a regression test against pinned fixtures.
- **SC-009**: For a representative image-scan fixture, a single `mikebom sbom scan` invocation that requests both CycloneDX and SPDX 2.3 output completes in measurably less wall-clock time than two sequential single-format invocations against the same target (target: at least 30% reduction, reflecting the fact that the deep-hash and layer-walk work is performed once instead of twice), and the produced CycloneDX and SPDX files are bit-identical to those produced by the two separate invocations.

## Assumptions

- **Serialization scope**: SPDX 2.3 output in this milestone is JSON only. Tag-value, YAML, RDF/XML, and RDF/Turtle serializations are out of scope; they can be added in a follow-up without disturbing this milestone's internal model.
- **CLI pattern**: Format selection follows the existing `mikebom sbom scan` noun-verb surface with a format flag on `scan` (matching the existing `--format` surface); no new top-level noun is introduced. Per FR-004 (clarified in this session), the format selector accepts a list so a single invocation can emit multiple formats from one scan; the existing default (CycloneDX, single file) is unchanged when the user passes no format.
- **Ecosystem coverage parity**: SPDX output covers exactly the same ecosystems CycloneDX covers today (apk, cargo, deb, gem, golang, maven, npm, pip, rpm) on day one. Adding new ecosystems later is a separate milestone and must update both serializers in lockstep.
- **Schema and validator availability**: The SPDX 2.3 JSON schema is treated as the contract. Schema validation in CI uses a maintained pure-Rust JSON Schema validator (`jsonschema = "0.46"` or successor) against the vendored SPDX 2.3 JSON schema. Cross-checking against an external SPDX-tools validator (e.g., LF `pyspdxtools`) is acceptable as a one-off pre-release smoke test but is NOT required in CI — reintroducing a Python or JVM runtime dependency in CI is explicitly out of scope per `research.md` R5.
- **SPDX 3 target**: The SPDX 3 stub emitter (FR-019a) targets the latest stable SPDX 3.x JSON schema available at the time the milestone is planned; the specific minor version (e.g., 3.0.x vs 3.1.x) and the choice of ecosystem covered by the stub are pinned by the implementation plan, not this spec. The data-placement map's SPDX 3 column is written to be forward-compatible with subsequent SPDX 3.x minor versions; concrete profile choices outside the stub's scope may be marked "defer until SPDX 3 profile X stabilizes."
- **VEX handling**: CycloneDX `vulnerabilities[]` has no direct SPDX 2.3 home. Per FR-016a (clarified in this session), VEX is emitted as an OpenVEX JSON sidecar file (`mikebom.openvex.json` by default) referenced from the SPDX document. OpenVEX is chosen over CSAF VEX for this milestone because it is a single-file JSON spec designed for exactly this use case and is the lowest-friction match for the existing CycloneDX `vulnerabilities[]` payload; CSAF VEX support, if requested later, would be additive.
- **Annotations as fallback**: Per FR-016, `mikebom:*` properties, evidence, and compositions land in SPDX `annotations[]` as mikebom-namespaced JSON-encoded comments. The annotation namespace, comment schema, and target placement (document vs. Package level) are documented in the data-placement map.
- **No change to scan pipeline**: This milestone adds a serializer and documentation only. It does not change scan, resolution, ecosystem parsers, or the internal `ResolvedComponent` model beyond what is strictly required to make the CycloneDX path format-neutral (if any changes there are needed at all).
- **No change to CycloneDX output**: CycloneDX output after this milestone is byte-identical to before it for any given scan. This is enforced by test (SC-006), not just intent.
- **Constitution compliance**: The milestone follows the project's standard pre-PR verification (`cargo +stable clippy --workspace --all-targets` and `cargo +stable test --workspace` both clean), per `CLAUDE.md`.
